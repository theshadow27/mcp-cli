**Investigation gate (#2103 → #2138):**
- The mandatory pre-impl nerd-snipe (per `references/investigations.md`) cost ~$4.03 / 68 turns of opus 4.7 reading test bodies, CI artifacts (`bun-test-logs-26082570102-1`), and the full git history (8 prior fix attempts: #1105, #1071, #1433, #1552, #1934, #1980, #1987 → #2103). Findings posted as issue comment with timing math:
  - **Confirmed mechanism**: `killPid` skipped because `childPid === null`. The 30ms CI failure timing rules out timing-window hypothesis — any path through `killPid` takes ≥110ms. Childpid is null because `transport.pid` reads after an event-loop yield in `ensureConnected`, during which the close handler can fire and `_process = undefined`.
  - **Secondary bug found**: assertion null-handling blind spot — when `originalStartTime === null` (ps failed under CI load) the conditional reports "alive" regardless of actual state.
  - **Test fix landed** (#2138): poll-with-deadline using `isAlive(pid)` (5s deadline, 100ms poll); production race deferred to #2135 per investigations.md "don't bundle production fix into test repair".
- This is the kind of outcome the gate exists for. Previous sprints (#1934, #1980, #1987) shipped 8 different fix attempts to the symptom — each treated the wrong mechanism because nobody made it find root cause first.

## What worked well

- **Investigation gate paid off.** #2103 spent ~$4 on opus root-cause work that produced a concrete mechanism (childPid race) AND a test fix (#2138) AND a production-side follow-up (#2135). Compare to sprint 47/#1870, where the gate was skipped and opus produced a CI-retry workaround that "hits on every sprint PR." This is the second sprint in a row where a flaky gate produced more than just the test fix — investigations.md is paying for itself.
- **Sonnet parallelism.** 14 sonnet impl/QA sessions ran concurrently against quota=100% (user-authorized overage). The 5h block reset happened mid-sprint; sessions kept running on the daemon's `extraUsage isEnabled` credits with no perceptible interruption.
- **Reviewer self-repair scales to 17 findings.** PR #2136 went through 3 review rounds (review-bot + Copilot rounds 2 + 3 = 10 + 7 + 5 = 22 findings total; 9 fixed in code, 13 deferred to followup issues). Each round was a `mcx claude send <reviewer>` rather than a fresh opus repair — the reviewer kept the worktree/PR context across rounds.
- **Triage-then-spawn-fresh-QA worked on Copilot late arrivals.** Multiple PRs (#2118, #2120, #2125, #2126, #2132, #2134, #2138, #2139) had Copilot post inline comments AFTER the initial QA verdict (sometimes 60–180s after the push). The pattern: if QA already wrote `qa:fail` for a stale reason, bye it and spawn a fresh one with the comment-IDs as input; if QA hasn't run yet, just send the work to the QA worker as input. No false-positive merges shipped.

## What didn't work

- **`session.permission_request` events spam the monitor stream for Edit/Write even when `--allow Edit Write` is set.** The opus #2023 impl session fired ~15 false-positive permission_request events in 5 minutes for tools that were already allowed. Approving via `mcx claude approve` returned `Error: No pending permission requests` immediately — the requests resolve via the allowlist before reaching the pending queue, but the event still fires. Orchestrator learns to ignore Edit requests, which means real Edit prompts (rare but they exist) would also be dropped. **Filed #2129.** Workaround applied: ignore Edit-tool permission_request events from sessions that were spawned with Edit in --allow.
- **investigations.md spawn template missing `--allow Bash`.** The canonical spawn shape at `references/investigations.md:62-87` doesn't include a permission allowlist, so the first #2103 investigation spawn immediately hit 7+ Bash permission walls and burned ~$0.50 of opus thinking before I bye'd + respawned with `--allow Read Glob Grep Write Edit Bash`. **Filed #2121.** Concrete fix: update the doc template to include the allowlist.
- **Worker tried to commit a meta-file edit (`.claude/skills/sprint/references/retro.md`) inside the #1945 impl PR.** Caught by orchestrator inspection before push. Interrupted, sent worker the meta-file rule from `run.md`, worker reverted the hunk and filed #2131 (meta label) for the proposed retro.md addition. The rule worked: meta changes did NOT leak into the sprint PR. Cost: one extra orchestrator turn and a follow-up meta issue.
- **ctx.gh `done.ts` statusCheckRollup adapter regression manifested mid-wind-down.** After `mcx phase install` reloaded `done.ts` to the post-#2023 ctx.gh-based version, the daemon was still running the pre-#2023 binary (no ctx.gh wired into AliasContext). `mcx phase run done` threw `undefined is not an object (evaluating 'ctx.gh.pr')`. Daemon restart fixed the binary, but the *new* done.ts also reports `ci_not_green` on PRs whose `statusCheckRollup` is all SUCCESS — because the new code uses REST `/check-runs` while the old code used GraphQL `statusCheckRollup` which has different semantics. **Filed #2148.** Workaround for #1563 merge: `gh pr merge 2139 --squash --auto` + `work_items_update phase=done --force`. The deferred follow-up #2147 already covers the underlying issue.
- **`sleep N` in orchestrator was lazy polling.** User caught me using `sleep 60 && check status` between event checks despite having a Monitor stream open that pushes events as they fire. The Monitor stream already delivers `session.idle`, `pr.merge_state_changed`, `ci.finished` notifications — sleeping between them is wasted cache. Behavior corrected mid-sprint.
- **#2114 worker briefly stalled at full-test-suite run.** Tokens froze at 4199 for ~90s with `lastResult: "Now run lint:"`. Could have been the test suite legitimately running (~100s) or a hung Bash, but the worker's nudge response showed it had progressed past lint and was on the test results — most likely just the long-running test suite. No action needed.
- **macos-13 runner queue.** The first manual workflow_dispatch from main (acceptance criterion 3 for #2092) saw macos-14 + macos-15 run in 2-3 min, but macos-13 was still `queued` at sprint close (36+ min). This is GitHub Actions hosted-runner availability, not a code problem. Will land async.

## Patterns established

- **Three-round adversarial review on goal-heavy PRs is fine.** PR #2136 went through 3 rounds (initial review + Copilot rounds 2 and 3) and the reviewer-self-repair flow handled all of them without spawning opus repair. The round-cap (review ≤ 2) is intended for the *adversarial reviewer* re-spawning; Copilot adds extra rounds that aren't counted. Don't apply the cap to Copilot-driven rounds.
- **At quota=100% with `extraUsage isEnabled: true`, treat quota as informational not blocking.** The `available: true` flag matters more than the utilization number when overage credits exist. Sprint 57 ran 14 sonnet sessions + 2 opus through the 100% reset window with user authorization; no throttling needed.
- **Phase-run skipping `--dry-run` correctly.** Per the existing memory `feedback_phase_run_no_dry_run`, all phase ticks in this sprint went `mcx phase run <phase> --work-item #N` without `--dry-run`. No false advance, no missed state writes. Memory is load-bearing.
