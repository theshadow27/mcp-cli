### P0 regression discovered: idle timeout kills sessions on restart (#712) (sessions d135795f + ac5de159)

Sprint 12 started normally but within minutes the daemon was killing all spawned sessions. Investigation revealed a cycle: daemon idle-times out, restarts, orphan reaper kills all sessions from the previous instance, daemon starts with 0 tracked sessions, idle fires again in 5 minutes. Repeating every ~5 min all afternoon (visible in logs as 10+ restart cycles).

**Root cause**: `reapOrphanedSessions()` in `index.ts` runs at startup BEFORE `ClaudeServer` is created. It calls `db.endSession()` on all non-ended sessions and SIGTERMs their processes. Then `restoreActiveSessions()` runs and finds nothing to restore. Empty `activeSessions` → idle timer fires → daemon shuts down → cycle repeats.

This was a regression introduced by PR #683 (orphan reaper, v0.4.1). Before #618 (well-known WS port), daemon restarts lost sessions anyway so the reaper's unconditional kill was invisible. #618 made restart-survival a real scenario and the reaper became the blocker.

**Secondary issue discovered**: even if the reaper were fixed, `restoreActiveSessions()` never called `this.onActivity?.()` — so idle timer wouldn't reset even if sessions were properly restored.

**Third issue**: `stopDaemon()` in `ipc-client.ts` sleeps 200ms then deletes the PID file and socket while the old daemon is still shutting down. `ensureDaemon()` on the next `mcx` call sees no PID file and spawns a new daemon. Old daemon finishes, deletes its socket; new daemon crashed on EEXIST earlier. Race creates two daemons.

**Fourth issue**: zombie daemon accumulation. Two Tuesday daemon instances were still listening on the socket file descriptor. Kernel round-robins `accept()` across all listeners. mcpctl's IPC polls were being distributed across 4 daemons — the current one might only see 1-in-4 requests, not nearly enough to keep the 5-min idle timer alive.

**PR #714** (session ac5de159, run interactively by user due to the bug bootstrapping problem):
1. `orphan-reaper.ts` — reaper no longer kills processes. Only ends DB records for sessions whose processes are dead/recycled. Alive sessions preserved for `restoreActiveSessions()`.
2. `claude-server.ts` — `restoreActiveSessions()` now calls `this.onActivity?.()` after restoring sessions.
3. `ipc-client.ts` / `stopDaemon()` — waits for process exit (10×100ms poll) instead of fixed 200ms sleep before deleting socket/PID.
4. `ws-server.ts` — WS port retry window increased to 10×500ms (was 5×200ms) to handle socket linger after old daemon exits.
5. `index.ts` — kills process recorded in PID file on startup, cleans stale socket file. Prevents zombie accumulation.
6. `check-coverage.ts` — added `index.spec.ts` to timing exclusions (spawns 13 daemon instances, inherently >5s).
7. Added `resetIdleTimer` diagnostic logging: `Idle timer fired (Xs since last reset)`.

Filed #713 (separate): `mcx shutdown --force` when active sessions exist.

Also filed #715: mcpctl uptime counter shows too many decimal places (`4m 11.067863000000017s`).

## What worked well

1. **User ran the fix session directly** — the bootstrapping problem (sessions die before implementing the fix) was solved by having the user run `claude` in a terminal tab. The right tool for the right situation.
2. **Log forensics pinpointed the bug precisely** — timestamp analysis of daemon logs identified the exact sequence: reaper kills sessions → restore finds nothing → idle fires. No guessing.
3. **`MCP_DAEMON_TIMEOUT=3600000` workaround** — extending the idle timeout was the right interim mitigation to unblock the sprint.
4. **Orphaned sessions completed before dying** — the 6 sessions lost mid-sprint-11 had already pushed their PRs. Work survived the crash.
5. **PR triage accuracy** — all 13 sprint 11 PRs routed correctly (low → QA only, high → adversarial → repair → QA). Zero broken merges.
6. **Spike-then-plan discipline for agent providers** — investigating OpenCode before filing issues caught the Crush fork confusion, the FSL license issue, and the ACP multi-vendor standard. Issues filed were buildable from day one.

## What didn't work

1. **Zombie daemon accumulation was invisible** — 4 daemon processes sharing one socket went undetected until `lsof` on the socket file. `mcx status` only shows the PID-file daemon. The kernel round-robining requests across all 4 daemons made mcpctl's keepalive polls effectively useless. No tooling surfaced this.
2. **Pre-flight daemon check skipped** — sprint 12 started without verifying the daemon was on the latest build. Sprint run reference now has an explicit pre-flight step.
3. **P0 mid-sprint spike broke the batch cadence** — 4 PRs from Batch 1 were open with no QA when the spike was declared. #138 and #695 lost their worktrees with zero commits. Had to re-spawn them.
4. **`stopDaemon()` 200ms sleep was never enough** — 200ms against a daemon shutting down 5+ virtual servers and workers was always a race. The sleep has been there for multiple sprints unexamined. Would have caught it earlier with a liveness poll.
5. **Dual-start race creates cascading instability** — even after fixing the orphan reaper and the stopDaemon race, the well-known WS port can still be unavailable if the old daemon hasn't released it yet. Port falls back to random → session can't reconnect → idle timer fires → another restart. Stable only after a clean single-daemon run.

## Patterns established

- **Bootstrap problem for daemon bugs**: If the fix requires a spawned session and the daemon kills sessions, the user must run the fix session directly in a terminal tab. Don't fight the bug with the broken tool.
- **`lsof` the socket, not just `mcx status`**: `lsof /path/to/mcpd.sock` reveals all processes listening. Multiple entries = zombie accumulation. `mcx status` only reflects the PID-file process.
- **Reaper contract**: Orphan reaper should only end sessions with dead processes. If `isProcessAlive(pid, startTime)` returns true, leave it alone — `restoreActiveSessions()` will pick it up. Unconditional `endSession()` on startup is wrong.
- **`onActivity()` after restore**: Any code path that populates `activeSessions` must call `onActivity()` to reset the idle timer. The timer doesn't know about restored sessions unless explicitly notified.
- **Spike before epic decomposition for new protocols**: validate transport/handshake/approval/token-tracking works from Bun before filing implementation issues. Spike findings go on the issue as comments — implementer reads them and starts coding.
- **Codex/OpenCode/ACP as parallel tracks, single shared permission layer**: `packages/permissions/` is the critical shared piece. Build it once, both tracks use it. Don't duplicate the `allowedTools` matching logic.
