Caught at QA round 1, fixed in repair, re-QA'd green. Good end-to-end dogfood.

Orchestrator unblocked #1749 as soon as #1748 pushed (not merged), saving ~5 minutes of serial wait. Edges are a planner hypothesis, not an invariant — verify against actual PR files.

- **Already-fixed close-outs took <$0.50 each** (#1692 at $0.34, #1697 at $0.13). Fast verify-and-close flow saved ~$6 vs attempting reimplementation.

## What didn't work

- **#1714 needed 4 QA rounds** — most of any issue this sprint. The failure cascade was: (R1) 7 review findings; (R2) QA flagged 4 Copilot threads + real race condition; (R3) QA flagged 6 threads + red CI (TS2559 + Bun coverage crash); (R4) rebase needed — `main` merged #1760 meanwhile, which added a test using the old `createWaitForEvent(fn)` single-arg API the PR was changing. The opus repair in R3 introduced a `process.kill(pid, undefined)` bug because `process.once("SIGINT")` handlers receive no args, so the shared `onSignal(sig)` always got `undefined`. Lesson: when wiring signal handlers with a shared callback, use per-signal wrapper closures that capture the signal name explicitly — not a parameter the event doesn't supply. The final rebase was a one-line fix; took one more QA round to verify. Total cost: ~$15 on #1714.
- **#1714 touching `phase.ts` wasn't in the sprint plan's hot-shared file watch.** Plan listed `#1745, #1747` on `phase.ts` (serialized by dependency edge) but #1714 also touched `phase.ts` (SIGINT wiring in `runPhase`/`executePhase`) and was not serialized. No merge conflict actually happened, but the rebase onto main (post-#1760 merge) was the conflict that forced round 4. Plan's file watch should be regenerated from Explore's file-level read of candidate PRs, not just grep'd from issue titles.
- **`mcx claude bye` prints "Error: Removed worktree:" / "Error: Deleted branch:"** — noisy and misleading (labeled Error, even on success). Sibling of #1749 (which we just fixed). File follow-up: stderr label should match severity (info vs error).
- **Release-commit push blocked by autoapprover** (confirmed by user) — `git push origin main` for `release: v1.7.4` was denied because the autoapprover treats direct-to-main as always-suspicious, but release commits are the one legitimate case. **Follow-up to decide in sprint 45 planning:** either (a) cut releases via a PR (`release/vX.Y.Z` branch → auto-merge) so the autoapprover is satisfied, or (b) teach the autoapprover to allow commits whose subject starts with `release: v` on main. Option (a) adds ~2 min of CI wait; option (b) needs autoapprover config we may not own. User preference so far: "probably ok to do a PR."
- **#1585 needed 2 repair rounds** despite reviewer self-repair on round 1. Round 2 QA surfaced 4 new Copilot threads on the repair commit itself (zero-threshold validation, wrong-tool error attribution, missing try/catch in timer callback, streaming-tool false positives). The repair commit triggers its own Copilot review cycle — reviewer self-repair is not "one-and-done"; it extends the QA surface. Cost: ~$10 on #1585 vs original budget.
- **#1518 initial scrutiny was `high`** (review phase) but the reviewer found 2 🔴 that needed opus for the dead-code-path redesign. Then reviewer round 2 found 2 more 🟡 on the opus repair. Help-infrastructure refactors spread across dispatcher/provider/log-injection surfaces; "contained" is easy to miscall. Cost: ~$10 on #1518.
- **Phase state transitions required manual orchestrator work** beyond the plan: every repair → qa and repair → repair cycle needed `phase_state_delete` on the stale `session_id` before the new spawn, otherwise the phase handler returned `in-flight` for the dead session. `#1745`'s fix auto-persists phase but doesn't clear phase-scoped state. File follow-up: `mcx phase run` should optionally clear phase-scoped state on transition-into (opt-in flag on the phase definition).

## Patterns established

- **Release-commit-via-PR** is the likely sprint-45 fix for the autoapprover block. Log the rationale in the retro so the next planner has context.
- **Reviewer self-repair is round-scoped, not incident-scoped**: each repair commit triggers a fresh Copilot-review pass, so the QA gate on unreplied threads (sprint 43 meta fix, closes #1716) will surface *new* threads on the self-repair commit. Budget for one more repair round beyond the self-repair one.
- **Close-as-already-done flow** is cheap and should be the default first action when a candidate issue is "obvious" — check `git log` for a fix keyword match before spawning impl. Takes <$0.50 vs ~$2 for even the smallest re-implementation.
- **Planner's hot-shared file watch should read PR diffs, not issue titles**: sprint-plan's `worktree-shim.ts` prediction for #1748 was wrong (fix landed in `gc.ts`), #1714's `phase.ts` touch was invisible. Use Explore + `gh pr diff` previews (where a PR already exists for a sibling issue) when building the serialization graph.
