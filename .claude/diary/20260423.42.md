## What worked well

- **Reviewer self-repair pattern** — used 6 times (#1574, #1576, #1580, #1592, #1600, #1626, #1577). Each saved a fresh opus repair session (~$10-15 each). When the review finds 1-3 blockers with file:line + concrete fixes, sending the reviewer `send` back beats spawning a new session that has to re-load the worktree + PR context.
- **#1621 landing mid-sprint** was the dominant productivity unlock. First 4 merges needed `--admin`; all subsequent merges used plain `gh pr merge` on CLEAN state. Planning sprints around "land the infra unblock first" paid off.
- **Parallel impl fan-out** peaked at 17 concurrent sessions without a cost blow-up. 6-way parallel impl at once on Batch 2 worked cleanly because serialization was only on shared files (ipc-server.ts), not on feature graphs.
- **Triage decisions were accurate** — low-scrutiny PRs all went qa→done; high-scrutiny PRs all needed at least one review round. The 120-churn / 100-addition thresholds in triage.ts correlated well with actual complexity.
- **CoalescingPublisher landed before consumers** — #1577 and #1581 could start in Batch 2 without waiting for #1574 to merge. Sprint 41's "drip Batch 2 opus only as Batch 1 finishes" rule was safe to relax once #1574's branch was pushed.

## What didn't work

- **Copilot-thread fatigue, again.** Same pattern as sprint 41: almost every PR got qa:fail round 1 with "N unresolved Copilot inline comments." QA sessions cleared them via `send` → push → swap label, burning ~20-30 min per PR unnecessarily. The adversarial-review/implement skills should pre-reply to Copilot threads before declaring "push complete." Fix: implement skill should run `gh api /pulls/{n}/comments` after push, and either reply or include a dismissal in the PR description. Filed #1707 for this.
- **I checked out a PR branch on main by accident.** While byeing a session, my main checkout ended up on `fix/issue-1600-site-worker-restart` — 3 phantom commits ahead of origin/main. The #1425 sprint-active guard didn't catch it because the commits existed on the PR branch too (not a real phantom). Safe to just switch back to main, but it's the exact class of bug #1425 was supposed to prevent. Root cause: session bye cleanup left main checkout pointing at the byed session's branch. Needs a post-bye invariant check.
- **`mcx phase run qa --work-item '#N'` doesn't update work_items.phase.** Every transition (impl→triage, triage→qa, review→qa, etc.) required a manual `work_items_update {phase:"qa", force:true, forceReason:"..."}` after `mcx phase run` returned. The `approved: X → Y` output in the phase-run stdout is theater — phase state doesn't actually flip. This is a real bug; orchestrator had to force-update every transition, ~40 force calls across the sprint. Needs a fix: phase run should persist phase on success.
- **qa.ts phase script emits `action: wait` when label not set yet** — this blocked #1577 round 2 until I realized the phase script was saying "wait" but my follow-through was spawning a session anyway. The phase script's "wait" return should be surfaced more prominently in the orchestrator loop.
- **Pre-commit hook reported `1 fail / 0 test identifier`** on the v1.7.2 release commit (and passed on retry). The test-failures log only has March entries — the failure was a flaky `bun test --coverage` run but nothing was captured. Filed #1708 for instrumentation.
- **Meta-file drift during sprint** — .claude/phases/* got re-hashed mid-sprint after #1685 merged, breaking every subsequent `mcx phase run` until `mcx phase install`. The orchestrator-rule about meta files being sprint + retro only held, but the phase lockfile regeneration should be automatic on drift detection, not a manual recovery step.
- **`mcx phase run qa --from=<>` flag is mandatory for repair→qa and qa→repair transitions** — without it, the phase script uses the last transition log entry as the "from" and rejects. Needs either auto-detection or a clearer error.
- **I blocked spawning #1597 at 79% quota** when reset was 7 min away — user corrected: "quota thresholds are for early/mid block, fire for effect near reset." Saved as feedback memory. The 80% freeze rule in run.md is too literal; should check `resetsAt` before blocking.

## Patterns established

- **"Send reviewer back for self-repair when 🔴s are contained"** — now a documented win. Budget this into adversarial-review's workflow: the reviewer has the worktree + PR context already loaded, fresh opus repair sessions cost $10-15 more.
- **Orchestrator-initiated Copilot-thread sweeps** — when a QA posts qa:pass but API shows unreplied top-level Copilot comments, `send` the same QA to post replies rather than spawning a repair. Sprint 42 used this for #1580 (4 threads resolved via existing session).
- **Admin-merge unblock as sprint gate** — when an orchestration blocker lands (#1621 for sprint 42, similar for future), merge order should prioritize it. Subsequent merges use CLEAN + plain `gh pr merge`, halving manual intervention.
- **Quota end-of-block override** — `feedback_quota_end_of_block.md` added to memory. Near reset, fire for effect.
- **Bonus picks during sprint** — user pulled #1695 in mid-sprint; it landed cleanly. Low-risk wildcard-matching, 50-line sonnet impl. Pattern: small, well-scoped issues filed during the sprint can be squeezed in without re-planning if quota permits.
