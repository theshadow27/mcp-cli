## What worked well

- **Adversarial self-repair pattern held up.** When Frank-the-reviewer posted ⚠️ Changes Requested on #1424, `mcx claude send` back to Bob-the-reviewer to fix the contained findings saved spawning a fresh opus repair. Same pattern for QA fails with Alice/Carol on #1440 and Eve on #1450. ~$15–20 avoided.
- **Single-pointer `update-branch` cascade** (adopted late, captured as new memory `feedback_sprint_bulk_and_cascade.md`). First half of the sprint I broadcast-updated all BEHIND PRs after each merge → each branch ran CI ~5× (N² CI load). Switching to "update only the next-to-merge" dropped CI runs to O(N). User flagged both this and the `for`-loop approval cost directly.
- **QA caught real defects.** 3/8 QA sessions returned `qa:fail` with genuine issues: #1419 grep pattern collision (`^ 0 fail$` matched on real failures), #1433 broad `catch{}` swallowing non-ESRCH errors, #1424 TOCTOU in update path. All fixed before merge.
- **Bonus contrib drive-through** (#1454). Light-touch review with "user manually tested, classify Copilot comments as blocker/defect/noise" was the right scope for a large PR that had been hand-verified. Merged cleanly.

## Patterns established

- **Single-pointer merge cascade.** When N PRs are auto-merge armed and BEHIND after a merge, update-branch ONE (the next-to-merge), let auto-merge land it, repeat. N total CI runs instead of N². Captured in `feedback_sprint_bulk_and_cascade.md`.
- **Bulk jq for status rollups.** `gh pr list --json ... --jq '.[] | select(...)'` replaces `for pr in ...; do gh pr view $pr; done`. Every `for` loop triggers a manual approval — bulk query is one approval.
- **Light-touch review for user-tested contribs.** For large PRs the user has hand-verified, skip full adversarial review; audit the Copilot comments for real blockers only. Merge if 0 blockers found.
- **Mid-sprint plan additions OK with explicit authorization.** User authorized 2 temp slots for #1442/#1443 when their urgency became obvious (#1425 recurrence). Both landed. Process: target count got bumped in sprint file header with a note explaining why.
