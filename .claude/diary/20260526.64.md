## What worked well

- **User's "do #2397 first" call was decisive.** The diff-aware gate cut the
  pre-push hook to changed files only. Every subsequent worker's local
  `am-i-done --pre-push` ran in 6-7s instead of 60s+; the orchestrator's docs-
  only timestamp push would have needed `--no-verify` for every meta edit
  without it, and the impl/repair iteration loops collapsed accordingly. The
  decision to require adversarial review on #2397 surfaced the
  short-circuit-test-proves-nothing finding that would have otherwise shipped
  silently.
- **QA-self-repair pattern**: twice, QA missed a substantive Copilot inline
  finding (#2253 vacuous-pass; #2370 probeSocket 503) and posted qa:pass; both
  times sending QA back with "Copilot found X at file:line, your verdict
  missed it, fix it" produced a clean self-repair in 1-2 turns + reverdict.
  Cheaper than spawning a fresh repair session.
- **Reviewer self-repair when scope is small (≤6 contained findings)**: #2397
  (6 findings) and #2396 (4 findings + 2 from Copilot) both stayed in scope
  and produced clean ✅ verdicts in one repair turn each. #2391 (6 findings,
  but blocker #1 required cross-file routing dispatch redesign) correctly
  routed to fresh opus repair — no convergence-failure churn.
- **The "pre-paid investigation" callout in the plan** prevented re-spawning
  nerd-snipes for #2330/#2383/#2388/#2253. Workers got a one-paragraph
  mechanism brief in the impl prompt and went straight to the fix. The one
  miss (#2330 implementer misreading the rule) was caught in QA, not in impl.
- **#2400 and #2388 fix bundling**: I filed #2400 (sandbox escape) at sprint
  start as a P1, the #2391 reviewer independently found the same evidence
  (empty `Test <test@test.com>` commits) on the grok PR's branch, and the
  #2388 worker accepted my "both likely solvable in one PR" briefing and
  delivered `gitSafeEnv` covering both. Single PR, two issues closed.

## What didn't work

  (merged 2026-05-25) were both already done when the sprint planned them.
  Workers cost ~$0.24 each to discover this and untrack. Fix: planner should
  query `gh issue list --state closed --search "<title-keywords>"` for each
  candidate before adding to the table. **File issue for plan.md addition.**
- **Stash@{0} content diverged from plan.** Plan said it held an old sprint-63
  run-state doc; it actually held 55 lines of `automation-dispatcher.spec.ts`
  test work. Did NOT drop. Plan's "save lesson then `git stash drop`" assumed
  contents stable from planning to run, but stashes shift. Saved the lesson
  (`feedback_dont_end_on_passive_wait.md`) without the drop. Fix: planner
  should `git stash show stash@{0} --stat` and quote the file list, not just
  recall "what was in it."
- **The flaky #2388 blocked my docs-only sprint-meta timestamp commit.** Hook
  doc explicitly sanctions `--no-verify` for WIP, but the auto-classifier
  blocked it and I had to ask. User authorized; got through. Lesson: when the
  issue being fixed IS the gate blocker, the orchestrator's path is the
  diff-aware gate's first user — #2397 making this collapse afterward.
- **Cross-PR cascade noise on every merge.** Every main merge triggered
  ~5-10 `pr.merge_state_changed` UNKNOWN→CLEAN/BLOCKED events across all
  in-flight PRs as the daemon's poller re-checked them. Mostly OK noise, but
  some impl workers got rebase nudges that produced churn (worker tries to
  rebase, hits conflict, asks for help). Going forward: maybe debounce
  cascade events per-PR until the actual local CI has fired.
- **#2391 had a `Test <test@test.com>` empty commit on its branch already**
  — same mechanism as #2400 — which means the same env-leak bug had been
  silently corrupting PR histories for an unknown duration before this
  sprint. The squash-merge of the sprint PRs makes this invisible after
  merge but it's still uncomfortable. Going forward: `gh pr view --json
  commits` audit step in the QA prompt for any commit author not in the
  team list — would catch this proactively.

## Patterns established

- **Adversarial review on every in-flight PR, even when CI is green and the
  PR is the orchestrator's own.** The orchestrator persona authoring #2397
  did not exempt it. The review caught a behavioral test that proved nothing
  (short-circuit was "verified" by a fake-bun that returned the same failure
  on both passes) — a finding that would have shipped silently otherwise.
- **For repairs the orchestrator can describe in 1-3 file:line citations
  with concrete code, `send` the existing session back; for redesigns or
  cross-file architecture, spawn fresh opus.** The cost differential is
  ~5x ($0.4 self-repair vs $2 fresh repair), and the choice is determined
  by *what kind of edit* is needed — see #2391 (cross-file dispatch table
  → fresh repair) vs #2397 (4 contained changes → self-repair).
