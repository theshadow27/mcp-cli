## What worked well

1. **Multi-method QA for untestable renames** — the rename QA session (691e3bf4) used 5 verification methods in parallel (source grep, binary string search, compiled binary exercises, CI artifact names, install script). Caught 20 files the implementation missed. For changes that can't be unit-tested, parallel audit agents are the right tool.
2. **Real integration tests beat mocks for Workers** — the `alias-server.ts` coverage problem seemed hard (Worker spawning, MCP handshake, etc.) but timing it showed 126ms. Writing a real integration test got to 100% coverage and revealed an actual bug (control message handler being overwritten by `WorkerServerTransport.start()`).
3. **WorkerTransport as a reusable joint** — implementing the MCP `Transport` interface over `postMessage` is ~30 lines and makes any Worker a first-class MCP server. `ServerConnection` is the universal abstraction; zero special-casing in `ServerPool`.
4. **`gh issue edit` in a loop** for bulk issue text updates — clean approach, no need for local file staging.

## What didn't work

1. **`replace_all` only catches exact matches** — in the rename, `Usage: mcp logs` was fixed but the continuation line `mcp logs --daemon` was missed because the pattern didn't match the second line. Text-replacement renames require an explicit full-file search after each pass.
2. **Writing-plans skill invoked unexpectedly** — a deactivated plugin skill got triggered mid-session during #103 implementation. Interrupted workflow. Plugin deactivation state should be respected (filed if it isn't already).
3. **Old `mcp` binary left stale lock/socket** — when the user ran the old binary during the protocol mismatch chaos, it created a stale lock file. `ensureDaemon` then hit the full 5-second `DAEMON_START_TIMEOUT_MS` before recovering. This surfaced the `isProcessMcpd` repeated `ps` spawn issue.
4. **IPC server tests required `null` aliasServer arg** — adding a new optional constructor param to `IpcServer` forced updating every `new IpcServer(...)` call in tests. Small friction but repeated across many test cases.

## Patterns established

- **Rename QA: grep source, then grep binary, then exercise commands** — three passes minimum for any rename. Source grep finds most; binary string search catches what minification might hide; exercising commands catches missing help text continuations.
- **Cached PID verification for fast polling** — `isProcessMcpd()` now caches the verified PID. No repeated `ps` spawns in `waitForDaemon` poll loop.
- **Coverage ratchet is enforced, workarounds are blocked** — attempted to add an exclusion for `alias-server.ts`, caught by the CLAUDE.md rule, timed the real integration test (126ms), wrote the real test instead. The ratchet held.
- **Strategic issue updates before implementation** — for #136, checked the post-#103 state before implementation began and updated the issue to reflect that the virtual server infrastructure already existed, removing the "supersedes #103" framing and simplifying the implementation scope.

## Architecture insight filed this session

The `mcx serve` / `MCP_TOOLS` concept (#136) crystallized: by exposing the daemon as a stdio MCP server with a curated tool filter, project `.mcp.json` files can give Claude Code access to exactly the tools it needs for that project. Claude's fine-tuning around `mcp__server__tool` call patterns becomes an asset instead of a liability. The rename (#130) and alias virtual server (#103) are prerequisites that make this work cleanly — `mcx serve` will be ~100 lines sitting on top of the full daemon infrastructure.
