### Complexity estimation research (session a4d5957f)
Built a full data-driven complexity estimation system, then invalidated its own premise:

**What was built:**
- `score.ts` — TypeScript AST scorer using `ts.createSourceFile()`. Counts LOC, branching nodes, imports, max nesting depth. Two modes: `--modify` (existing files being changed) and `--analog` (structurally similar to new code being written)
- `db.ts` / `backfill.ts` — SQLite schema + script that mined all 103 merged PRs into the DB with AST metrics, per-file additions/deletions, and extracted symbol lists
- `validate.ts` — Tested 6 ML approaches (kNN, gradient boosting, ordinal regression, calibrated analogy, with/without log-transform, with/without embeddings)
- `validate-triage.ts` — Binary classification validation on historical PR data

**What the data said:**
- Best pre-implementation predictor: 29.7% exact tier accuracy (barely above 19.8% "always guess median" baseline). Spearman ρ ≤ 0.49 for all structural features
- Text embeddings made predictions *worse* — 145 MAE vs 119 MAE
- `srcAdditions >= 107` post-implementation triage: 97.2% F1 on historical data
- `srcChurn >= 120 OR riskCount >= 2` post-implementation: 92.5% F1, 0% false negatives

**The insight that killed the original approach:** The user realized: always implement with opus (cheap relative to review), then triage based on the actual diff to decide how much validation to do. Measuring after is nearly perfect; predicting before is essentially noise.

**New pipeline in `/run-issue`:** implement (always opus) → triage.ts reads actual `git diff` → low path (QA only) or high path (adversarial-review → repair loop → QA)

**Files committed to main:** `.claude/skills/estimate/{score.ts,db.ts,backfill.ts,validate.ts,validate-triage.ts,triage.ts,embed.ts,SKILL.md}`

## What worked well

1. **Full review cycle (implement → adversarial → repair → QA)** caught real bugs every time — adversarial reviews found blocking issues in 8/10 PRs
2. **Reusing review sessions for repair** — send repair instructions to the same session instead of spawning new; saves context reload
3. **`mcx claude wait --timeout`** is much better than `sleep` polling — interruptible and event-driven
4. **4 parallel sessions** is the sweet spot; 5+ hits rate limits and stuck-connecting issues
5. **Data-driven validation for the estimator** — actually running validate.ts against 103 historical PRs killed a bad approach early and surfaced a better one (post-implementation triage). The experiment was worth running even though the conclusion was "don't do this pre-implementation"
6. **Parallel agent dispatch for diary backfill** — 7 Sonnet agents in parallel, all wrote good entries. The JSONL extraction script was the hard part; analysis was fast

## What didn't work

1. **`session:result` from `wait` is ambiguous** — means "idle" (waiting for input), not "ended." Killed a working stats-tab session (`be0ae248`) by calling `bye` after seeing the event. Had to `spawn --resume be0ae248...` to get it back — but `mcx claude resume` silently ignores `--resume` flag (it only works on `spawn`). Filed #441
2. **`mcx claude resume` doesn't restore conversation history** — `resumeWorktree` calls `claude_prompt` with `{ prompt, cwd }` only; never passes `sessionId`. The `--resume` flag on `mcx claude resume` is silently ignored. The resumed session sees git state only, not conversation history
3. **Triage script returns 0 for everything** — `triage.ts` when run from main (not the feature worktree) shows 0 src churn because it's comparing against the same branch. Known issue, workaround: run from worktree or fetch the branch first. Treated all PRs as high scrutiny until fixed
4. **`/clear` + send in rapid succession** loses the second message — need to wait for clear to settle (get the idle event) before sending the next instruction
5. **Complexity prediction is fundamentally noisy** — Spearman ρ ≤ 0.49, 29.7% exact tier accuracy even with gradient boosting and log-transform. Embeddings made it worse. This is an inherent property of the problem at this sample size, not a fixable feature engineering issue
6. **Daemon crashed mid-session** (the EPIPE bug #411 was fixing) — lost 2 active sessions mid-sprint. After rebuilding with the fix, no more crashes
7. **Stale daemon instances from parallel sessions** polluted the QA environment — pkilling everything was too aggressive (hit ~30 sessions); careful cleanup with process inspection is required

## Patterns established

- **Post-implementation triage, not pre-implementation estimation**: `triage.ts` reads `git diff --stat` on the actual branch after implementation. `srcAdditions >= 107` catches 97.2% of high-scrutiny PRs. Pre-implementation complexity scoring is noise
- **Session state check before acting on `wait` result**: After `wait` returns, check session state before calling `bye`. `session:result` = idle (needs follow-up), not ended
- **Review sessions fix directly**: After adversarial review flags issues, send repair instructions to the same session. Saves spawning a new session and reloading context
- **Wait for `/clear` to settle**: Send `/clear`, wait for idle event, then send next instruction
- **Diary backfill from JSONL**: `extract-sessions.ts` extracts user+assistant text from `~/.claude/projects/` JSONL; filter by user message count >= 5 (not file size); spawn one Sonnet agent per date
- **File issues with reproduction data**: Bare "this happened" issues are useless; include commands, logs, timestamps, and context
- **`mcx claude spawn --resume <session-id>`** is the correct way to restore conversation history (not `mcx claude resume`)
