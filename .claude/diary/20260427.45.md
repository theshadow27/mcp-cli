## Patterns established

- **Override triage to adversarial review when the plan calls high scrutiny.** Triage looks at diff size; high scrutiny is about *consequence*, not *size*. A 36-LOC change to memory-safety logic deserves opus review; a 200-LOC test file rewrite typically doesn't. The orchestrator should treat the plan's scrutiny column as authoritative, not as a hint.
- **Reviewer self-repair extends to multi-finding pushes.** Sprint 33-34 used self-repair for 1â€“3 contained findings. This sprint: Eve handled 5 findings on #1589 (including a full ring-buffer rewrite) â€” and 3 findings on #1738 (including a structural reorder of cursor-advance vs processing). The "1â€“3 contained" guideline can relax: the deciding factor is *whether the diagnosis is already loaded*, not finding count. If the reviewer's findings are concrete (file:line + fix description) and don't require multi-file architectural judgment, send for self-repair regardless of count.
- **Release via PR branch (sprint 44 retro proposal) is the right call.** `release/v1.7.5` â†’ auto-merge PR â†’ tag at merged-sha worked exactly as designed. The autoapprover never fired, the release commit went through the same gate as everything else, the tag landed clean. Cost was ~2 min of CI on the release PR â€” cheaper than the per-sprint user round-trip the autoapprover used to require.
- **Background-task `until` polls + TaskOutput with high block-timeout** is the right idiom for "wait until merged" during wind-down. No 270s cache-window math required when the runtime auto-notifies.
