The actual fix landed mid-sprint as part of #2147 (`checks()` now augments REST with commit statuses); pre-fix workaround was direct merge. No new issue — the fix shipped.
- **#2143 merged with stale `qa:fail` label.** Round-2 QA flagged flaky CI on `server-pool.spec.ts` (unrelated to PR changes; "Re-trigger CI to confirm"); CI cleared on `gh run rerun --failed`; orchestrator armed auto-merge without flipping the PR label. Merge judgement was fair (impl was already verified clean) but the label hygiene was wrong — anything downstream keying on the literal label sees `qa:fail` in main's history. Filed #2177 + new memory `feedback_qa_label_hygiene.md`.
- **Server-pool flakiness** — even after sprint 57's #2138 (poll-with-deadline) and this sprint's #2142 (`pollUntil` conversion), still timing out on PR CI under load (filed #2176). #2100 tracks the remaining 136 `Bun.sleep` violations; #2142 chipped 2 off.

## Patterns established

- **Speculative parallel spawning of blockedBy tasks** — when the dependency is a file/region overlap (not a logical schema change), spawning dependents while the anchor is still in QA is safe so long as the orchestrator broadcasts rebase on merge. The user's "use more slots" nudge can become a default heuristic when quota headroom is high (<30% sonnet utilization).
- **QA label hygiene** — when merging on flaky-CI-cleared rerun, flip the PR label first; don't merge on stale `qa:fail`. New memory `feedback_qa_label_hygiene.md` + #2177 captures the rule.
