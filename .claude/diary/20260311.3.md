# 2026-03-11 Night — Sprint 10 Retro

> First sprint using the new 4-phase skill (plan/run/review/retro). v0.4.0 released.

## What shipped

11 PRs merged, 4 issues closed as already-fixed. 15/15 planned issues resolved.

**Features**: mail viewer TUI, `mcx auth` UX, Python-repr parser, `_metrics` virtual server, release automation.
**Fixes**: transcript polling, pager cleanup, viewport scroll.
**Internal**: pollUntil async, command handler tests, test-failure-log SQLite migration.

## What worked

1. **Aggressive backfilling**: When sessions completed early (4 issues already fixed), immediately spawned Batch 2/3 replacements. This kept all 5 opus slots saturated.

2. **Triage → scrutiny routing**: Low-scrutiny PRs went straight to QA (4-6 min turnaround), high-scrutiny got adversarial review. Saved review budget for where it mattered.

3. **4-phase skill structure**: Plan produced a good issue list with batch assignments. Run phase was mechanical — follow the batch order, triage, route. Review phase was a single command. Having the phases explicit in the skill made the orchestration less improvisational.

4. **Adversarial reviews catching real bugs**: Every high-scrutiny PR had blockers found. The `--dry-run` bug in release.ts, the infinite loop in python-repr, the expired-token dead code in auth — all would have shipped without review.

## What didn't work

1. **ExitPlanMode not in --allow list**: Session 732292bf (#466) burned $0.58 in a permission loop. The `/implement` skill enters plan mode, but ExitPlanMode wasn't whitelisted. Fixed in run.md (#654).

2. **Worktree explosion**: 48 worktrees created for 15 issues (113 total accumulated). Every repair/review/QA phase created a new worktree instead of reusing the implement one. Root causes: (a) `bye` doesn't clean up daemon-created worktrees because `cwd: null` (#671), (b) the skill used `--worktree` everywhere instead of `--cwd`. Fixed the skill (#670); cleanup bug filed separately.

3. **#409 needed 2 repair passes**: The first repair session claimed to fix the bugs but pushed to a different branch name. The second review found the original branch still had the old code. Repair sessions need explicit branch targeting.

4. **#129 also needed 2 repair passes**: Round 1 review found escape handling bugs. Round 1 repair fixed those but introduced a new infinite loop on bare `+`/`-`. The repair→review→repair loop adds latency but catches real issues.

5. **Codex integration broken (#666)**: `mcx codex spawn` fails with serialization error. Couldn't use Codex for reviews as planned. P1 for next sprint.

## Patterns learned

- **"Already fixed" issues are free wins**: 4 of 15 issues were already resolved by prior work. The implement session just verified and closed them. Better to plan conservatively and close fast than to miss things.

- **High-scrutiny repair loops are expensive but justified**: The 5 high-scrutiny PRs averaged 2.5 phases each (review + repair, sometimes x2). Total cost is higher but the code that ships is substantially better.

- **Branch naming matters for repairs**: When spawning repair sessions, always pass `--branch <exact-branch-name>` to avoid the repair landing on a new branch.

- **Triage script is reliable**: Low-scrutiny predictions matched reality — small PRs shipped cleanly. High-scrutiny predictions also matched — every flagged PR had real issues.

- **Skill self-improvement works**: The sprint found 2 bugs in its own skill (ExitPlanMode permissions, worktree reuse) and fixed them mid-sprint. The skill is better for the next sprint than it was at the start of this one.
