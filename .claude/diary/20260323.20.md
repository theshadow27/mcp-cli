# Sprint 20 Retro — 2026-03-23

## What happened

Sprint 20 was planned March 19 targeting the unified agent command epic (#906). 12 of 15 issues shipped on day 1 before a quota exhaustion incident killed the sprint mid-flight. The sprint sat in an indeterminate state for 4 days with 2 zombie sessions accumulating $519 in costs. Re-planned March 23 with 14 issues (3 carryover + 11 new). All 14 merged in a single session.

## The quota incident

A parallel swarm on another project burned through Claude credits. The mcp-cli sprint kept running oblivious, burned through extra usage, then hit a hard wall. Two sessions from March 19 sat zombie for 4 days — one with uncommitted work, one with merge conflicts. Neither was visible as problematic because `mcx claude ls` doesn't show session age.

**Root cause:** No awareness of quota state, no pause mechanism, no session age visibility.

**Filed:** #968 (pause/resume epic), #962 (show session age in ls).

## The stale daemon bug

The daemon from March 19 (0.10.0) was still running when we rebuilt (0.11.0) on March 23. This caused:
- `mcx claude wait --short` returning raw JSON instead of compact format
- Session costs showing as 0/$—
- `--timeout` on `wait` being ignored (2.5 min instead of 30s)

We couldn't restart because sessions were active. The mismatch was only discovered when session stats looked wrong.

**Filed:** #964 (stale daemon breaks --short and zeroes stats).

**Lesson:** The daemon version warning exists but is easy to miss. Pre-flight should verify version match, not just `mcx status`.

## What worked

- **14/14 merged in one session.** Re-plan was clean and execution was smooth once started.
- **Adversarial reviews caught real bugs.** #910 (resume shim) went through 3 review+repair rounds. Every round found genuine issues: hardcoded `main` branch, catch-all swallowing errors, `--wait` no-op, greedy `--allow` parsing, missing SIGKILL escalation. Without adversarial review this code would have shipped broken.
- **Saturating the pipeline.** 5 opus impl + unlimited sonnet review/QA kept throughput high. Batch 2 started backfilling as batch 1 finished.
- **Filing issues during the sprint.** 5 follow-up issues filed (#946, #950, #962, #963, #964) — problems that would've been invisible without the "file everything" rule.

## What didn't work

- **#910 needed 3 repair rounds ($12.77 + $3.59 + $4.17 impl, plus $4.37 QA).** The resume shim is inherently complex (600+ lines) but the repair cycle was inefficient. Each repair introduced new issues because the repairer didn't have full context of prior reviews. Consider: sending the full review history to repair sessions, not just the latest findings.
- **#913 also needed 2 repair rounds.** The `--resume` flag rejection and `--compact` regression were introduced by the first repair. Same pattern — repair without full context.
- **Stale daemon was invisible for too long.** The warning only shows in `mcx status` and `mcx claude ls` (non-short). During monitoring I used `wait --short` exclusively and never saw it.
- **`wait --timeout` was ignored by stale daemon.** I escalated timeouts to 5 min, then 10 min, compounding the problem. Should have investigated why 30s wasn't working instead of working around it.
- **Python one-liner for JSON parsing.** Caught by user — should use `jq` or the tool's own formatting, not ad-hoc Python.

## Patterns to keep

1. **Always verify daemon version in pre-flight.** Add `mcx version` check to sprint start.
2. **Run `mcx claude ls` (not `--short`) periodically to see warnings and session ages.**
3. **Send full review history to repair sessions.** Not just "fix these 3 things" but the entire context of what was already tried.
4. **Don't escalate timeouts blindly.** If a timeout isn't working, investigate the root cause.
