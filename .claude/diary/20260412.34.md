## What worked well

- **Micro-repair pattern (from sprint 33) is a huge win.** 5 of 6 review rounds in this sprint were resolved by the reviewer fixing its own 🔴s in-place instead of spawning fresh opus repair. Saved ~$15–20 of opus repair + worktree churn. PRs #1299, #1327, #1328, #1345, #1359 all used it. The reviewer-self-select pattern ("fix if contained, reply `needs opus repair` otherwise") let the reviewer decide instead of the orchestrator guessing.
- **Adversarial review caught substantive bugs in #1299.** The NaN-bypass in `done.ts` (`Number.parseInt("") → NaN`, `Number.isFinite(NaN) → false`, merge proceeds on empty CI output) would have been a production-risk footgun in the v1.5.0 gate itself. Reviewer earned its cost just on that.
- **Triage `--pr N` (no worktree) worked everywhere.** Didn't have to reuse impl worktrees for scrutiny decisions. Faster, less state management.
- **Event-driven `mcx claude wait --timeout 270000`** — no sleep polling, caches stayed warm on most idle cycles.

## What didn't work

- **N² rebase cascade.** 15 PRs branched off main simultaneously, then serialized through merge queue. Each merge invalidated ~10 others → manual rebase worker spawns × ~5 rounds before user suggested the fix. Root cause: orchestrator didn't treat merge-train capacity as a constraint during spawn. **Fix applied this sprint**: user proposed and we spawned a dedicated sonnet "release-train engineer" session (`85194daa`) that serially rebases → waits for CI → lets auto-merge fire, then processes next. **Fix for next sprint**: spawn the train at sprint start, hand PRs to it via `mcx claude send` as each hits qa:pass. Pipelined instead of batched.
- **QA false-negatives from stale branches.** #1346 and #1330 both QA-failed because the branch predated #1316's EACCES fix on main. QA correctly flagged the revert, but the blocker was purely "branch out of date." Forced an extra rebase-repair-QA cycle per PR. Preventative: orchestrator should rebase-before-QA when main has advanced since impl time. (Better yet: the release-train handles this naturally when it runs continuously.)
- **#1338's `--no-verify` bypass (surfaced via #1356 investigation).** Sprint 33 shipped a lint violation to main because an implementer used `--no-verify` to work around #1337 (GIT_DIR test leak). Branch protection was not yet enabled. Now it is, but `enforce_admins: false` means admin-scope automation could still bypass. **Retro topic:** wire a PreToolUse hook in `.claude/settings.json` that blocks `git commit|push --no-verify` outright, or set `enforce_admins: true`. See sprint-35 plan.
- **#1299 scope-check stall.** Dogfood worker stopped at ~15 turns asking for scope confirmation on a plan-sized issue — reasonable behavior, but orchestrator had to actively nudge with explicit "proceed full scope." Bigger issues may benefit from a "full scope approved" line in the initial prompt when the plan has already scoped them.
- **Task-tool reminders spam.** System injected "use TaskCreate" reminders ~20 times across the sprint even though I was actively using them. Low signal. (Not a sprint-34 fix, but worth noting.)
- **Bun segfault on #1317 (Carol).** Known #1004 class. Worker handled correctly (opened bun.report URL per memory rule).
- **Copilot code review comments silently ignored across the sprint.** PR #1380 (#1299 dogfood) has **17 inline review comments from `copilot-pull-request-reviewer[bot]`**, none of which were acknowledged, resolved, or addressed. They went to `GET /repos/.../pulls/N/comments` (inline surface, file:line scoped) while the adversarial-review / repair / QA sessions only check `gh pr view N --comments` (PR-body surface). Same split for any human inline reviews that might land via the UI. Substance was real — one example: Copilot flagged `docs/phases.md:56` for marking `version` as required when the manifest loader defaults it (`z.literal(1).default(1)`). That's a doc-vs-code contradiction that would have confused future readers. User is enabling `required_review_thread_resolution` on the `main` ruleset so unresolved threads block merge, but the skills still need to teach all three phase agents (reviewer / repairer / QA) to enumerate **all four** PR-comment surfaces before labeling a PR ready:
  - PR body comments: `gh pr view N --comments` (and `--json comments`)
  - Inline review comments: `gh api repos/OWNER/REPO/pulls/N/comments` (file:line scoped, where Copilot lives)
  - Review containers: `gh api repos/OWNER/REPO/pulls/N/reviews` (top-level review summaries from each reviewer, state = `APPROVED | CHANGES_REQUESTED | COMMENTED`)
  - Linked-issue comments: `gh issue view $ISSUE --comments` (the issue the PR `fixes`, not the PR itself — sometimes discussion lands there)

  Each thread needs to be either (a) addressed with code change + reply, or (b) explicitly dismissed via `gh api .../comments/{id} -X PATCH` or resolving the thread via GraphQL. Labeling `qa:pass` with open threads is the bug.
- **Force-push rebases silently invalidate GitHub auto-merge.** After the mergemaster's `gh pr update-branch <N> --rebase` force-pushes, the PR's `autoMergeRequest` becomes `null` on some configurations, even though the PR looks identical from a GitHub UI perspective. The train sat in an open-loop polling #1389 for ~13 min (CLEAN/green) waiting for a merge that would never fire. Detected by the mergemaster's own escalation ("previous PRs merged within 1–2 min of CLEAN; this one hasn't in 13 min"). Fix: `mergemaster.md` now checks `autoMergeRequest` at the start of each PR's cycle and re-runs `gh pr merge <N> --squash --delete-branch --auto` if null. Idempotent; fires exactly when needed. The lesson is also a case for the "run the same check twice" defensive pattern — force-push changes server state in ways that aren't visible from the `gh pr view` default shape.
- **`qa:pass` and `qa:fail` can both be attached to the same PR simultaneously.** Sprint 33's PR #1303 merged with both labels — happy path was qa:fail → repair → qa:pass, but the transition only added the new label without removing the old one. Orchestrator's merge gate checks "does `qa:pass` exist?" not "is `qa:pass` the *only* QA label." Fix folded into the #1380 repair: the `qa.ts` / `done.ts` phase scripts must transactionally swap labels (`--add-label qa:pass --remove-label qa:fail` in one call, and vice versa on the fail transition). Add a small test for the swap. Post-merge, the QA skill docs should state this invariant explicitly so future phase-script rewrites don't regress it.

## Patterns established

- **Orchestrator headroom ≠ capacity to absorb more work.** Post-phase-rewrite, the per-PR orchestrator cost drops significantly (phase scripts return typed decisions instead of prose-driven bookkeeping), which opens real headroom. The tempting interpretation is "great, orchestrator can drive the merge queue too, retire the mergemaster." The better interpretation: **use the headroom for quality, not quantity of tasks.** Sprint 34's "idle between waits" cycles weren't slack — they were neglect windows where the 17 Copilot comments, the dual-label bug on #1303, the stale `qa:fail` label recurrence, and the auto-merge invalidation on #1389 all accumulated unnoticed. Filling those cycles with merge bookkeeping would have turned "not noticing problems" into "noticing fewer problems because the attention went to rebasing." Better uses for the headroom: proactive scans of all in-flight PRs for open review threads each cycle (not just at QA), session observability (watch for cost-runaway / stuck sessions earlier), sprint-level strategic reassessment (are we on track? should we cut scope? start next-sprint planning mid-sprint?). The mergemaster offloads **busywork**, not capacity — a concurrency primitive that lets the orchestrator stay attention-rich for the parts that need judgment. Keep the mergemaster; consider a `mergeStrategy: orchestrator | dedicated` config for small sprints (≤5 PRs, no parallel impl) where the ceremony isn't worth it.
- **Mergemaster was a discovery tool, not the production answer.** The sonnet session worked because it let us find the shape of the problem — auto-merge invalidation, force-push races, upstream blockers, queue ordering under pressure. But once the shape is known, "put a 200B-param model in a poll loop to call `gh pr update-branch` every 4 minutes" is the elevator-button problem. Production answer: a native `mcx merge-queue` service (Bun daemon, ~200 LOC, zero LLM cost) that reads the `work_items` table, ensures auto-merge is armed, rebases when BEHIND, polls CI, and sends `mcx mail` to the orchestrator on genuine failure. The mergemaster agent stays as an escape-hatch for the one weird PR per month where the service hits a case it can't resolve — but 99% of merges cost nothing. Alternative shape worth considering: merge-as-a-phase, where `done.ts` self-loops via `{action: "wait", reenterAfter: 240}` (needs phase runtime to support re-entry on a timer). File as sprint 35 goal candidate if user agrees. The existing `agents/mergemaster.md` should note "this is a fallback; the primary path is `mcx merge-queue` once that lands."
- **Release-train / "mergemaster" engineer (user-invented during this sprint — kept as escape hatch).** A long-running sonnet session whose sole job is: `gh pr update-branch --rebase` → poll CI → let auto-merge fire → next. No worktree, no local git ops, no `gh pr merge` calls (auto-merge already queued). Drains the QA backlog serially without burning opus cycles. **Next-sprint follow-through**: (a) enable GitHub's native merge queue (retires most of this work), (b) add `.claude/skills/sprint/references/mergemaster.md` describing the agent for the corner cases the queue doesn't handle — "most of the time unneeded, but when the queue stalls or kicks a PR back, a dedicated long-lived expert is cheaper than waking the orchestrator." Orchestrator can poke it for queue status or hand it PRs for manual intervention. (c) The pilot of both belongs to sprint 35, not sprint 34 — needs budgeted time to validate both layers.
- **Reviewer self-select for repair vs opus.** "If 1–3 contained fixes with your existing diagnosis, fix in place and update sticky. Otherwise reply `needs opus repair`." Let the reviewer decide. Worked 5/6 times.
- **Rebase-after-main-advance.** When QA fails purely because main moved forward (not a real bug), spawn a one-shot rebase worker, not a fresh opus repair.
