## The OAuth scope fix (#1076)

The most important fix this sprint. Atlassian (and likely other OIDC providers) returned HTTP 500 because mcpd sent no `scope` parameter during OAuth authorization. mcp-remote works because it defaults to `openid email profile`.

**First attempt was wrong.** Adding `scope: "openid email profile"` to `clientMetadata` passed the adversarial review's first blocker: `fetchToken()` reads `clientMetadata.scope` directly, so every token exchange would send the scope unconditionally, breaking non-OIDC servers. Dynamic client registration would also receive the scope, potentially causing registration failures.

**Second attempt was also wrong.** Moving scope to `getEffectiveScope()` with a fallback still defeated the SDK's own cascade — the SDK short-circuits when it receives a truthy `scope` parameter, so `scopes_supported` from discovery metadata would never be consulted.

**Third attempt got it right.** Return `undefined` from `getEffectiveScope()` when unconfigured, letting the SDK cascade run (config scope > resource metadata > discovery metadata). The `openid email profile` fallback only applies when the entire cascade fails. Per-server `scope` config field added for explicit override.

**Lesson:** OAuth is deceptively complex. A 2-line fix that looks correct can break servers with different auth models. The adversarial review process caught both regressions before they shipped.

## DX quick wins from session survey

The session survey (#1072-1075) surfaced high-frequency pain points:
- **220+ attempts** to use `mcx <server-name>` instead of `mcx call <server-name>`

All three were quick fixes with outsized impact.

## Pre-commit daemon test timeout

Every session in this sprint hit the daemon integration test timeout during pre-commit hooks. This is a systemic issue — the daemon tests take 120s+ but the pre-commit budget is 120s. Multiple sessions needed `--no-verify` approval. Filed as pre-existing (#1024 was supposed to fix this but the fix was insufficient).
