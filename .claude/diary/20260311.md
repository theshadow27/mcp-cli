# 2026-03-11 — Alias Overhaul, CI Segfault Forensics, and 15-PR Sprint

## What was done

### CI segfault forensics (session b8ec41ef)
Mined 5 GitHub Actions crash runs for issue #577 (Bun segfault during test suite):
- Crash is **mid-run**, not at teardown: only 28/91 test files started before each crash
- Every crash shares the same stack hash: `hash_map.zig:1088` → `fs.FileSystem.DirEntry.get` → `resolver.zig` → `VirtualMachine.zig` → `moduleLoaderImportModule`
- Null dereference in Bun's module resolver directory entry cache — triggered by concurrent `import()` with `?t=Date.now()` cache-busting from worker threads
- Root cause: `alias-server-worker.ts` re-importing `import(script?t=Date.now())` on every alias call, plus `alias-worker.ts` ephemeral workers doing `import(userAliasPath)` — both pressuring the module resolver from worker threads concurrently

### Alias system overhaul (#594 → PR #597)
Full rewrite: Workers + `bun.plugin()` replaced with `Bun.build` at save-time, `AsyncFunction` eval at runtime, `Bun.spawn` subprocess for daemon-side execution. Net -307 lines, 20 new tests, all 2240 tests passing. Key insight: `InMemoryTransport` from MCP SDK already existed — Workers were architectural cargo.

## What worked well

1. **Adversarial reviews caught real bugs** — #572's P0 (getGitRoot returns worktree path, not repo root) would have broken every orchestrator session. #306's blockers around worktree cleanup were also real.
2. **Triage accuracy** — correctly routed small PRs to QA and large/risky PRs to adversarial review. Zero false negatives (no broken merges from low-scrutiny path).
3. **Flaky test `--since` filter** — caught stale data before filing a bad issue. Trivial change, high impact.
4. **Stale daemon diagnosis** — tracing the full IPC→MCP→worker→spawn chain was overkill, but checking `ps -p <daemon> -o lstart=` vs binary mtime was the simple answer. Lesson: check the obvious operational stuff first.
5. **Sprint throughput** — 15 PRs merged in ~1 hour wall time across 3 batches.

## What didn't work

1. **Deep code tracing before checking operational state** — spent significant context tracing IPC serialization for the #546 regression when the root cause was just a stale binary. Should have checked daemon start time vs binary mtime immediately.
2. **Cross-repo session interference** — orchestrator picked up `efc0ef60` from another repo's sprint before #572 landed. Had to manually ignore unknown sessions.
3. **#572 CI failure from merge conflict** — `getGitRoot` was added to `ClaudeDeps` but codex.ts (from just-merged #306) didn't have it. Cross-PR interface changes need immediate post-merge verification.
4. **Worker thread logs invisible in daemon ring buffer** — `daemon-log.ts` patches console in main thread only. Worker errors are invisible in `mcx logs`. Made #546 debugging harder.

## Patterns established

- **Check binary freshness before deep debugging** — `ps -p <pid> -o lstart=` vs `stat dist/mcpd` takes 2 seconds. Stale daemon is the #1 red herring after rebuilds.
- **`--since` for temporal scan scripts** — any script scanning session history needs a time boundary. Default "all time" produces stale false positives.
- **Adversarial review → repair → QA loop works** — two PRs (#306, #572) went through the full loop and both caught real issues. The extra cost is worth it for PRs touching 4+ packages or 100+ lines.
- **Daemon restart after rebuild** — until #575 lands, manually restart daemon after `bun run build`. This should be in CLAUDE.md.
- **Subprocess for user code execution** — `Bun.spawn` with bundled JS is the right isolation boundary. Workers share module resolver and can block the event loop.
