## What worked well

- **Re-plan-and-document pattern.** Pre-flight exposed that v1.5.0 shipped a phase pipeline the CLI couldn't invoke (`--dry-run` stripped, non-dry-run is a transition recorder not an executor). Rather than trying to run the original plan, wrote an explicit re-plan to `.claude/sprints/sprint-35.md` with "RE-PLANNED" notice + "Stage A → Stage B → Stage C" gated structure. Took ~30 min, saved the sprint. User's instinct to include legacy-spawn fallback in the original plan was the only reason `/sprint run` could even start.
- **Sequential Stage A** (A1 #1396 → A2 #1349 → A3 #1381) was the right call. A1 was a ~1-line fix; without it A2/A3 couldn't be verified. Parallelizing them (as the original plan did) would have wasted opus cycles.
- **Stage B explicit validation checkpoint.** Single issue, manual transitions through real `mcx phase run`. Produced unambiguous pass signal — "handler emitted JSON at every phase, orchestrator piped, state persisted." Without this step, Stage C's success could have been attributed to legacy spawn momentum.
- **Adversarial-review self-repair for contained findings.** #1407's reviewer correctly self-selected "needs opus repair" for the design trade-off (transition-log ordering) — which is exactly the signal the pattern is meant to surface. Saved ~$10–15 on micro-repairs elsewhere (#1401 reviewer fixed its own 2 Copilot findings).
- **Triage-to-QA routing for low-scrutiny PRs.** Everything marked low (`#1378`, `#1377`, `#1384`, `#1403`, `#1418`) passed QA first try. High-scrutiny (`#1407`, 297 additions) correctly routed to review-first.
- **`mcx claude wait --timeout 270000 --short`** as the event loop worked every time. No spurious hangs; interruptible; stayed under the 5-min cache TTL.
- **Workers responded cleanly to `send` for plan approval** (#1388 worker paused for 8-file-scope confirmation; `send` unblocked it without re-spawn).

## What didn't work

- **Context rot in the prior orchestrator.** v1.5.0 was released as an RC with the assumption that the phase pipeline worked. The prior orchestrator asserted "done" on things that weren't (at ~345k tokens of context — well short of the 1M limit). This is a real failure mode: models degrade quality before they hit hard context limits. Mitigation this sprint: shorter-lived sessions, explicit pass/fail criteria for validation (Stage B), skeptical pre-flight before trusting any "this is ready" claim from a prior run. Long-term: need automated post-release smoke tests (e.g. `mcx phase run impl --dry-run --work-item <test>` must print JSON, fail CI otherwise).
- **Worker containment failures.** Two separate meta-file-adjacent commits landed on main from a worker's main-repo-root cwd (#1425). The worker escaped its worktree and then pushed bypassing branch protection via inherited admin bypass. Fix direction: workers should never have write access to main's cwd; also audit when/how workers acquire push credentials that bypass branch rules.
- **`mcx claude bye` deletes local branches labeled "merged" while remote PR is still open** (#1255 recurrence, hit at least 3 times this sprint during session cleanup). Cosmetic but confusing; harder when combined with stale "D .mcx.yaml" blocking worktree removal (from #1383 before it landed).
- **QA sessions went "disconnected" and ballooned past `session:result`.** Bob session `a613f83d` hit 111,962 output tokens for a trivial single-file QA before I `bye`d it. `disconnected` is undocumented state; the worker kept generating tokens after the qa:pass verdict posted. Filed as #1426. Cost overrun: ~5x on that one session.
- **`mcx phase run triage` error opaque when `branch` missing on work item.** Error says "requires a work item with issueNumber and branch" — gives no hint which field is actually null. `work_items_update` doesn't auto-populate `branch` when you set `prNumber`. Hit it 3–4 times this sprint. Filed as #1424 with suggested fixes.
- **`/sprint review` release cut was botched on first attempt.** Ran `git commit` → pre-commit's lint hook rejected package.json formatting (biome wanted single-line arrays) → commit failed but tag got created (locally) before commit failure was noticed → pushed tag to origin → Release workflow triggered on old commit with `version: 1.5.0` in package.json → had to cancel workflow, delete draft release, delete tag local+remote, fix lint, re-commit, re-tag. Recovery was clean (no force-push) but the flow assumed `git commit && git tag && git push` is atomic. It isn't when the commit hook fails. **Fix direction for review.md**: run `bun lint` BEFORE `git commit`; run `git tag` AFTER commit succeeds; push tag only after push commits succeeds.
- **`git log` from a stale local main.** Twice during the sprint, my local main drifted (worker commits ended up in the main checkout; PR merges on GitHub weren't fast-forward-able locally). Had to `git reset --hard origin/main` with a branch-backup step to preserve worker commits. Related to containment failure above.
- **I `bye`d the impl session for #1396 immediately after PR was pushed, then had to spawn a fresh repair session when Copilot inline comments came in.** Lost the worktree + PR context that the impl session had. Per memory ("Don't `bye` spikes") — should wait for CI + auto-review to complete before `bye`. Fix direction: `bye` only after `qa:pass` AND zero open threads across all 4 comment surfaces.

## Patterns established

- **Gated-rollout pattern for skill re-validation.** Stage A (strict sequential prereqs, solo sessions) → Stage B (single-issue validation, orchestrator-driven, explicit pass/fail) → Stage C (aperture, parallel backlog, conditional on B). Generalizes to any "ship tooling + prove it works under load" sprint.
- **Re-plan when pre-flight invalidates assumptions.** Pre-flight isn't just a checklist — it's an opportunity to kill the sprint's premise before burning compute. Document the re-plan in the sprint file with "RE-PLANNED {date}" and the reason.
- **Orchestrator runs `git pull --ff-only` + checks for stray worker commits between phases.** Workers sometimes commit to main's cwd (#1425). Detecting early prevents `git log ... --oneline` from showing phantom commits at release time.
