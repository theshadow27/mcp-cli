**P1 daemon usability** (the headline arc):
- #1836 (PR #1886) — parallel `mcx claude spawn` race producing ghost
  sessions. Three root causes diagnosed by Alice: worktree name collision
  (`Date.now().toString(36)` not unique within a millisecond), `cwd: null`
  in session metadata when the spawn dies before sending `system/init`,
  and no cleanup on `spawnClaude` failure (sessions stay registered with
  no process). Fixed via `crypto.randomUUID().slice(0,8)` for names,
  pre-populating `state.cwd` from config, and a try/catch around
  `spawnClaude` that calls `removeUnspawnedSession` + posts `db:end`.
  Adversarial reviewer (also Alice in self-repair) caught that the
  ghost-cleanup test was at the wrong level — exercising
  `removeUnspawnedSession` directly instead of `handlePrompt`'s catch
  block. Test rewritten in self-repair commit.
- #1837 (PR #1895) — bye on a ghost session destroying an unrelated
  active worktree. Defensive scan added: before destroy, query the
  session table for any other session with the same `cwd`. Refuse and
  return a clear error if found. Reviewer caught a TOCTOU between the
  scan and the destroy (`this.sessions` snapshot vs concurrent bye);
  hardened in self-repair.

**Data correctness** (the under-the-hood arc):
- #1864 (PR #1879) — `updateWorkItem` was read-modify-write without a
  transaction. Wrapped in `db.transaction()` with optimistic locking
  (`version` column, migration v6, `WHERE version = $version`). Adversarial
  reviewer caught the real find: `upsertWorkItem` (also on the work-items
  hot path) bypassed the version bump in its `ON CONFLICT DO UPDATE`
  clause, leaving the same race open. Fixed inline. Migration v6
  ALTER+setSchemaVersion was also non-atomic (sibling pattern in v3, v4)
  — wrapped in `db.transaction()` in the same self-repair pass.
- #1859 (PR #1887) — applied `schema_versions` pattern to `StateDb`
  (replacing 15+ bare `try { ALTER TABLE } catch` blocks). Adversarial
  reviewer caught error-path untested. Self-repair added the missing
  tests.

## What worked well

- **Reviewer self-repair pattern saved real money.** All 4 high-scrutiny
  picks (#1836, #1837, #1864, #1859) had adversarial reviews that found
  real correctness gaps. In every case I sent the reviewer back with the
  micro-repair prompt instead of spawning a fresh opus. The reviewer
  already had the worktree, the diagnosis, and the file paths loaded.
  Fix-and-update-sticky cycle averaged ~$1 per PR vs. ~$3 for a fresh
  opus repair. Sprint-47 retro flagged this pattern; sprint 48 used it
  on 5 PRs and it held up every time. Promote to skill rule.
- **Plan-time issue audit caught #1798 before spawning.** The plan
  listed #1798 as a low-scrutiny pick. At run start the work-items table
  showed it tied to PR #1817 (already merged). Verifying state via
  `gh issue view` before spawning saved ~$0.50 + a wasted slot. Pattern:
  if `mcx tracked` shows a `prState=merged` for a planned issue, verify
  the issue is actually closed before spawning impl.
- **Pre-existing PR cross-links cleared via `work_items_update` with
  null.** #1818 and #1836 had stale PR links from a previous session
  (the resolver auto-found PRs that mentioned the issue number). Setting
  `prNumber: null, branch: null` via the `_work_items` API cleared them
  before spawn. Worth knowing — `mcx untrack && mcx track` doesn't reset
  these.
- **Zero admin-merges.** Sprint 47 retro called for ≤1 admin-merge as
  the leading indicator that the post-#1835 coverage CI fix held. Sprint
  48 used auto-merge on all 16 PRs with no overrides. The #1869 fix
  (mock-claude `--version` handler + `process.exitCode` switch) is
  holding.
- **Live demonstration of #1836 fix during the sprint.** I spawned 9
  sessions back-to-back at sprint start (Batch 1). All 9 came up alive,
  none were ghosts. The race the issue described is real but timing-
  sensitive — getting clean spawn-9 was the validation that the
  pre-existing fix attempt had landed (it had, in #1815/#1835 cluster);
  the explicit fix in #1836 was for the residual paths.
- **Live demonstration of #1837 fix during the sprint.** Carol's QA
  session for #1849 had uncommitted modifications when I tried to bye
  her. The bye correctly refused to destroy the worktree (the #1837
  guard landing post-merge mid-sprint), reported "Error: 1 modified".
  The session ended cleanly; only the worktree directory was preserved.
  Exactly the intended behavior.

## What didn't work

- **Copilot's per-PR thread firehose is the dominant cost driver.**
  Of 16 PRs, 14 needed at least one repair round. The blocker on most
  of them was "N unreplied Copilot inline threads" — not 🔴 correctness,
  but style/nits/false-positives. The QA contract treats unreplied
  threads as a hard fail, so each round meant a repair session that
  mostly *replied to threads* with "false positive" or "out of scope".
  Pattern: ~3 of 4 Copilot threads were nits; 1 of 4 had real value.
  Cost: estimated ~$8-12 of repair-session work was thread-reply
  ceremony. **Improvement for next sprint:** allow QA to dismiss
  obviously-out-of-scope Copilot threads inline (Windows PATH, "extract
  to const", "consider helper") with a one-line reply, rather than
  requiring a full repair round. The QA agents already write good
  dismissal replies — they just gate on the existence of unreplied
  threads, not their substance. This belongs in the qa skill, not the
  orchestrator. File a meta issue.
- **Pre-existing flaky #1882 (server-pool.spec.ts:1731 SIGTERM/SIGKILL
  race) showed up on three PRs.** QA agents flagged it correctly each
  time but had to wait for a CI retry. Cost: ~3-5 min per occurrence.
  Filed as #1882. Should be triaged into the next flaky-test mini-batch.
- **Sprint plan called for #1864 high-scrutiny review on opus, but the
  review phase script forced sonnet.** Both the impl and the review
  ran on sonnet (review phase doesn't read the plan's scrutiny column).
  Sonnet adversarial review still caught the upsertWorkItem version
  bypass, so this didn't bite. But for next sprint, make the review
  phase script accept a model override or read the plan's high-scrutiny
  list. File a follow-up.
- **Phase-script transition from triage→repair isn't allowed.** When a
  repair session completes and I tried to advance directly to qa, the
  phase manifest required going through qa first. Worked around with
  `--from qa` and a manual `phase=qa` write. Not a bug in the manifest
  (it's enforcing the state machine), but the orchestrator pattern of
  "after repair, re-run qa" needs to clear `repair_session_id` first
  *and* advance the phase, in that order. Captured as a recipe.
- **`mcx phase run repair --work-item ... --from qa` after I cleared
  `repair_session_id` re-spawned a NEW repair instead of advancing.**
  Bumped repair_round to 2. Had to rewind round=1, re-clear session_id,
  then advance to qa. Lesson: the orchestrator transitions phase by
  *writing `phase=qa`*, not by re-ticking repair after clearing. The
  repair phase is "if no session, spawn next round."
- **One manual rebase needed.** PR #1898 (#1608) had a merge conflict
  with #1896 (#1607) on `packages/command/src/commands/claude.ts` —
  both touched the dispatch table. Spawned a quick sonnet rebase
  session, ~$0.82, ~3 min. Predicted by the plan's "Hot-shared file
  watch". The chain serialization worked but the per-PR rebase still
  has to happen.

## Patterns established

- **Reviewer self-repair on contained findings** (1-3 file:line cited
  edits) is the dominant pattern for high-scrutiny PRs. Send the
  reviewer back with: "If contained, fix yourself. If multi-file
  judgment needed, reply 'needs opus repair'." Reviewer pushes a fix
  commit and updates the sticky to ✅. Worked on #1864, #1836, #1837,
  #1607, #1608. Promote to run.md.
- **Pull-in-replacement when a planned issue turns out already-done.**
  User asked mid-pipeline to "pull in replacements to keep the pipeline
  saturated" after #1798 was dropped. Picked #1848 (test infra) and
  #1812 (sites). Both were Excluded with reasons in the plan; both
  ended up shipping with one repair each. Lesson: the Excluded list is
  a usable backup pool when a slot opens, not a permanent decision.
