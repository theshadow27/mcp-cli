# 2026-03-08 — Reliability Diagnosis + Claude Session CLI

## What was done

### Timeout and stress test fixes (#170, branch: fix/170-timeout-mismatches, commit: bb660ee)
Diagnosed real-world reliability failures triggered by a "kick the tires" session that resorted to repeated `pkill`. Found two concrete bugs in `constants.ts`:
- **IPC/retry timeout mismatch**: worst-case retry budget was 135s (4 × 30s + backoff), but IPC timeout was 60s. The client aborted while the daemon was still mid-retry. Fixed by reducing `CONNECT_TIMEOUT_MS` 30s → 15s and `CONNECT_MAX_RETRIES` 3 → 2, making worst-case 48s — safely under the 60s IPC timeout. Added a comment documenting the budget constraint.
- **Health check ping timeout**: `PING_TIMEOUT_MS` 2s → 5s. The 2s timeout false-negatives under event loop pressure, causing the CLI to clean up the socket and try to start a duplicate daemon, leaving the original orphaned.

Also built stress test infrastructure:
- `test/slow-echo-server.ts` — configurable-delay MCP server
- `test/stress.spec.ts` — 6 tests: S1 (5 simultaneous mcx processes race to auto-start one daemon), S2 (20 concurrent CLI calls + 10 rapid sequential), S3 (fast server responds while slow server mid-call), S4 (daemon health after error bursts). Tests spawn real `mcx` processes — not direct socket calls — covering the full auto-start/lock/IPC path.

Changes across 7 files (+427 lines):

### #194 — Daemon idle timer counts WebSocket sessions (PR #195, merged)
Bug: idle timer only checked `inFlightCount` (IPC requests), not active Claude sessions. Daemon would shut down after 5min of IPC inactivity even with a live Claude session connected. Fix:
- `claude-server.ts`: added `activeSessions` Set, populated from existing `db:upsert`/`db:end` worker events; `hasActiveSessions()` method; cleared on `stop()`
- `ws-server.ts`: added `sessionCount` getter
- `index.ts`: idle timer defers shutdown while `hasActiveSessions()` is true
- 5 new tests; 1240 total passing

## What worked well

1. **Long-poll over existing HTTP IPC** worked cleanly for `mcx claude wait` — no SSE or WebSocket upgrade needed. Bun's HTTP server naturally holds the connection while the async handler awaits.
2. **Reusing existing waiter pattern** (`waitForResult`) for `waitForEvent` kept the implementation tight and well-tested.
3. **Stress tests using real mcx processes** (not direct socket calls) are significantly more valuable — they catch auto-start races, lock file contention, and CLI arg parsing issues that unit tests can't reach.
4. **Tracking session lifecycle from existing worker events** (`db:upsert`/`db:end`) for the idle timer required zero new inter-thread messages.

## What didn't work

1. **A Claude session killed the diagnostic session mid-analysis** (the "lol the other claude killed you" moment). The pkill habit of the tire-kicking session is a systemic risk — stale processes with no clean detection mechanism, so the "fix" nukes everything.
2. **IPC timeout budget wasn't documented**, so it had silently drifted out of sync with retry constants. Now has a comment, but it's still a fragile invariant maintained by convention.
3. One test initially counted `claude-server` tools as 6, failed when `claude_wait` brought it to 7. Simple fix but it's a brittle count assertion.

## Patterns established

- **Document timeout budget constraints inline**: when constants have implicit relationships (IPC timeout must exceed worst-case retry budget), add a comment at the definition site. Prevents future silent drift.
- **Stress tests at the process boundary**: spawn real CLI binaries in test, not just direct IPC socket calls. The extra coverage is worth the complexity.
- **Long-poll for async wait semantics**: no streaming infrastructure needed for event-driven waits — the existing HTTP IPC holds open cleanly.
