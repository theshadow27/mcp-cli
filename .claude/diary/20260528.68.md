## What worked well

- **Reviewer self-repair carried the load.** Most high-scrutiny PRs (#2540, #2533, #2541, #2542, #2549) had the adversarial reviewer fix its own contained findings rather than spawning a fresh opus repair. Cheap and kept context loaded.
- **The dependency chain executed cleanly.** Spec (#2540) → version negotiation/recording (#2541/#2542) gated on its merge; #2493 gated on #2541. blockedBy edges + reactive backfill kept the pipeline full without manual batch gating.
- **Triage routing was accurate.** Low-scrutiny/docs PRs → QA, high-churn → review, every time. The two overrides I made (route docs-spec #2540 and security fixes #2549/#2493 to *review* despite low churn) each caught real issues a QA-only path would have missed.
- **Adversarial review caught "looks done but isn't" bugs the impl + CI missed**: #2234 dead-code transport, #2533 fail-open, #2542 shared-file corruption, #2549 environment-dependent test. CI was green on all of them.

## What didn't work

- **I misread the rate-limit signal as a hard quota gate.** `quota_status` returned `utilization:100` and I declared a "full pause until reset." Wrong: the reading was 26 min **stale** (its `fetchedAt` frozen because the quota-*stats* endpoint itself was 429'ing), GitHub had full headroom, and the `[RATE LIMITED]` session flag is *soft backpressure* — sessions kept progressing through it (Bob completed a full review while flagged). The user caught it. **Fix:** check `fetchedAt` freshness and whether sessions show token deltas before treating utilization as a gate; `[RATE LIMITED]` ≠ blocked. (Memory: `feedback_quota_status_staleness`.)
- **#2331 was a meta-file violation that should never have entered the sprint.** "Add `mcx pr comments resolve` to impl/review prompts" sounded like a worker-prompt tweak; the prompts are actually built by `.claude/phases/*.ts` — orchestrator-only meta I read/run live every tick. The worker (correctly) implemented it there, producing an unmergeable-mid-sprint PR. Parked PR #2570, relabeled `meta`. **Fix:** plan.md must treat any issue whose surface is `.claude/phases/**`, `.mcx.yaml`, or `.claude/skills/**` as meta and exclude it at plan time. (Memory: `feedback_meta_issue_planning_guard`.)
- **I over-constrained #2234 and caused its headline blocker.** My "don't touch `*-session-worker.ts`" guardrail (meant to avoid the #2541/#2493 hot-file conflict) blocked the worker from wiring `resolveTransport()` into the production spawn path, so it shipped the feature as dead code. **Fix:** serialize conflicting issues with blockedBy (which I then did — #2541/#2542 after #2234) rather than forbidding a file the work genuinely needs; the conflict resolves by merge order, not by leaving code unwired.
- **Rate-limited workers got stuck in background-poll loops.** Eve (#2371) spent ~1.5h re-launching `am-i-done` as a background task, getting throttled before it could read the result, timing out, restarting. **Fix:** the unstick is to tell the worker to run `am-i-done` as a *foreground blocking* call (no `run_in_background`/`TaskOutput` polling) — broke the loop immediately on #2371 and #2542. (Memory: `feedback_foreground_am_i_done_unstick`.)
- **Zombie process CPU starvation (#2493 live incident).** `bye --keep-worktree` across impl→review→QA leaked `am-i-done`/`bun test` trees (the orphan-reaper only ran on daemon startup). 24+ accumulated, starving every session's validation (healthy am-i-done is ~104s; these hung 30-60+ min). Manually reaped (kill ppid==1 orphans + anything >20min), then #2493 shipped the permanent reap-on-bye fix.
- **#2543 churned 3 repair rounds** (review 2 blockers → QA-fail interrupt regression introduced by the fix → Copilot batch of 9). Stayed mechanical (typo, verdict-check, pollUntil, state-sync) — no design rethink needed, so not a /simplify case — but it was the most expensive convergence of the sprint.

## Patterns established

- **Foreground am-i-done to unstick rate-limited workers** — backgrounded validation + result-polling interacts pathologically with API backoff; a blocking foreground run sidesteps it.
- **Orchestrator adjudicates reviewer-vs-Copilot tradeoffs** instead of letting them ping-pong. On #2493, Copilot wanted `+D` (recursive, complete) and the reviewer chose `+d` (single-level, fast); I ruled `+d` correct (orphans run from worktree root; `+D` reintroduces the perf blocker) and had QA reply-and-dismiss rather than flip-flop into another repair round.
- **Route security/safety fixes to review even at low churn.** #2549 (22 lines) and #2493 (63 lines) both triaged low → QA, but both *kill/guard* things where a too-permissive "fix" is worse than the bug. The review overrides caught an environment-dependent test and the shared-worktree over-kill.
