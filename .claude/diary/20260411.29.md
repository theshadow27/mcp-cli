# 2026-04-11 — Clone, poller, event-driven orchestration (Sprint 29)

## What was done

5 PRs merged, v1.3.0 released. Three major features landed in one sprint.

**mcx clone (#1163):**
- Git-backed projection of Confluence spaces to local filesystem. 1022 pages in ~10s.
- Adversarial review found 3 P0 security issues: shell injection via page titles (execSync with template literals), CQL injection via user-supplied scope key, silent data loss when >250 pages changed.
- All fixed in one repair round. QA verified security fixes + full test suite.
- Gap analysis filed ~10 follow-up issues for remaining phases.

**Work item tracker Phase 2 — GitHub GraphQL poller (#1140/PR #1180):**
- Daemon polls GitHub for tracked PR state (CI, reviews, merges) on adaptive interval (30s active, 5min stable).
- Adversarial review found: no concurrency guard on poll(), in-flight poll can write to closed DB on shutdown, review status logic wrong (COMMENT after APPROVED flipped status back).
- All repaired.

**Work item tracker Phase 3 — event-driven orchestration:**
- `mcx wait --any` (#1141/PR #1186) — races session events + work item events. Adversarial review found unhandled rejection on Promise.race losers + 1 more red.
- Enriched `mcx claude ls` (#1142/PR #1185) — lifecycle line under each session showing `impl → PR → CI → QA → merged`.

**Also:**
- Triage fix (#1171/PR #1174) — triage.ts now works from PR number without worktree.
- Closed #1162 as duplicate of #1181 (bun test exit code 1 on Linux).

## What worked well

- **Adversarial review caught real security issues** — the clone PR had shell injection, CQL injection, and silent data loss. All 3 P0s. Without review, this would have shipped to production with exploitable vulnerabilities. The review system proved its value on the hardest test case: a large, unreviewed PR from an external session.
- **Parallel batching across dependency chains** — clone review + GraphQL poller impl ran simultaneously. Clone QA + triage fix ran simultaneously. Never idle.
- **Gap analysis as a distinct phase** — separating "what shipped" from "what's left" into a dedicated session produced a clean follow-up issue set. Better than having the orchestrator try to assess gaps.
- **CI fix → review → repair → QA pipeline** — the clone PR needed 4 phases (CI fix, adversarial review, repair, QA) and they flowed cleanly.

## What didn't work

- **QA sessions stuck polling CI** — both #1163 and #1180 QA sessions burned $4-6 each waiting for CI runs that were either already passing or queued behind segfault retries. The QA session should check for existing successful runs before waiting for new ones.
- **#1174 CI fix cost $3.46** — should have been a $0.50 session. The opus got into a testing loop instead of just fixing the build error and pushing. The CI failure was a 44-second typecheck error.
- **Clone CI fix didn't fully work first time** — the repair session pushed the security fixes which triggered a new CI run, and THAT one passed. The initial CI fix session ($2.87) was partially wasted because the branch needed more work anyway.

## Patterns established

- **Security review is non-negotiable for new packages** — the clone PR had 3 P0 security issues that were invisible to the implementer. Any PR that introduces a new package with external input (user-supplied strings, API responses as file content) must get adversarial review regardless of scrutiny triage.
- **Gap analysis after large PRs** — when a PR implements part of an epic, spawn a separate session to map implemented vs remaining and file follow-up issues. The orchestrator shouldn't try to do this while managing the pipeline.
