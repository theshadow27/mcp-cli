## What worked well

- **Triage model was accurate**: all low-scrutiny PRs (#1169, #1203, #1205, #1183, #1206, #1004) passed QA first try. All high-scrutiny PRs had real blockers found by adversarial review.
- **Two-round adversarial reviews caught regressions**: #1207 round 1 flagged 2 blockers, round 2 approved after fixes. #1170 round 1 found 10 issues, round 2 confirmed 8 fixed and caught a persistent `.gitignore` regression the repairer had dismissed as a false alarm.
- **Conversation pattern over dispatch**: When the #1208 reviewer misunderstood the bye-message purpose (thought it was for worker transcript, actually for orchestrator context through tool-call preservation), a single `send` redirecting the repair session produced an immediate acknowledgment and correct fix. Saved a full repair cycle.
- **Event-driven polling kept saturation high**: Maintained 5-10 active sessions throughout sprint. 5-minute poll cadence was the right tradeoff — low enough anxiety, high enough responsiveness.
- **QA found & fixed CI failures**: #1183 QA caught a missing `git config user.email` in `clone.ts` that broke CI; fixed inline before merging.

## What didn't work

- **Stale daemon caused 5 dead session spawns at sprint start**: After `bun run build`, spawned 5 sessions against a daemon still running the old binary. All showed `disconnected`. Had to shutdown+restart daemon and re-spawn. Cost: ~3 minutes and some confusion. **Fixes filed**: #1217 (pre-flight should restart daemon) and #1218 (P1 — spawn should refuse/warn on version mismatch).
- **Sprint docs referenced nonexistent `mcx wait`**: Run instructions use `mcx wait --any`, `mcx wait --pr`, but only `mcx claude wait` exists. Caught immediately when the first wait call failed. **Fix**: #1216 — replace all `mcx wait` references with `mcx claude wait`.
- **Phase machine rejects `impl → qa`**: Sprint docs describe `impl → qa → done` as a valid path for low scrutiny, but `work_items_update` errors with "Invalid phase transition." Workaround: transition `impl → review → qa` (two calls). **Fix**: #1227 — add `impl → qa` to phase machine; the docs are correct.
- **`.gitignore` regression across 4 branches** (#1207, #1170, #1210, #1166): All branches based on commits before `3e98548` had 1-line `.gitignore` vs 18-line main. Every reviewer flagged as merge blocker, every repairer initially dismissed as false alarm. Cost: 3 extra repair cycles. **Pattern fix**: when a sweeping commit lands on main mid-sprint, the orchestrator should issue a "merge latest main" directive to all active impl sessions before they push. Don't discover per-branch during review.
- **`bye --keep` broken in production path**: #1233 investigation revealed `--keep` only works in test path; production routes through `agent.ts` which has no `--keep` support. Updated issue with root cause for sprint 32.
- **#1240 session scoping leak**: phoenix-octovalve sprint could see mcp-cli sessions but not vice versa. Related bugs #1242 (null-repoRoot leak) and #1243 (repoRoot not recorded when core.bare=true) filed as good-neighbor critical for sprint 32.
- **#1250 wind-down rebuild hazard**: The wind-down `bun run build` + daemon restart can break concurrent sprints in other repos. Skipped daemon restart this time after user caught it. Wind-down needs to check for cross-repo sessions before rebuilding.
- **Branch count at 1,368**: Local branches accumulate ~15/sprint with no cleanup. Filed #1219 for `mcx gc` command with sprint wind-down integration.
- **Cumulative coverage drift caught only at release**: `packages/command/src/commands/vfs.ts` was at 67.5% line coverage on main when the v1.4.0 release commit hit the pre-commit hook, below the 80% per-file threshold. Each contributing PR (#1170 resilience layer, #1178 --depth flag) passed its own CI and review, but neither ran the per-file coverage gate against main's cumulative state. The two branches independently added code faster than #1203's test additions could keep up — individually fine, together they regressed. Same shape as the .gitignore-across-branches problem: merge-cumulative effects invisible to per-branch review. **Fixes**: (a) filed #1252 to make CI enforce the per-file coverage ratchet (pre-commit-only enforcement is a loophole anyone skipping the hook can drive through), (b) restored vfs.ts to 100% line coverage in the v1.4.0 release commit by exporting internal helpers with optional DI and adding 20 targeted tests for the resilience/error paths. Pattern to recognize next sprint: when two or more PRs in the same sprint touch the same file, add a coverage re-check on main after the second merge lands.

## Patterns established

- **File issues for every problem, including process bugs**: 8 issues filed during this sprint (#1216, #1217, #1218, #1219, #1227, #1240, #1250 + #1233 update). Pre-flight alone surfaced 3. Unfiled problems are invisible problems.
- **Redirect repair sessions when reviewers misunderstand intent**: The #1208 reviewer's "transcript entry destroyed on write" finding was technically correct but solving the wrong problem. A `send` explaining the correct purpose (orchestrator context via tool-call preservation, not worker transcript) got the right fix immediately. Cheaper than letting the bad fix ship and rolling back.
- **5-minute poll cadence for wind-down**: After batch 3 was in-flight, 5-minute polls were the right tradeoff. 30-second polling during mass completions, then 5-minute once the pipeline was steady.
- **Proactively rebase when sweeping commits land mid-sprint**: If a commit lands on main that affects all branches (like `.gitignore`, `.git-hooks`, shared config), broadcast to active impl sessions to rebase before push. Don't pay the cost per-branch during review.
