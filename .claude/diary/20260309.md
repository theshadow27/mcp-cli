## What worked well

1. **Adversarial review caught real design gaps** — PR #302 was closed entirely because review revealed the implementation delivered trace ID propagation but not span lifecycle. Saved merging code that would immediately be rewritten.
2. **Filing issues before implementing** — the sequence #70 → #290 → #293 → #309/#299 was planned deliberately, each unblocking the next.
3. **QA writing missing tests** — the QA session for #310 discovered zero span DB tests, wrote 16, and brought `state.ts` to 100% coverage before merge.
4. **Parallel adversarial agents** (eigenbot + pessimist-prime) consistently converge on the same P0 issues and provide different framings that catch different aspects of the same bug.

## What didn't work

1. **Merged PR #287 without testing the compiled binary** — `new URL()` pattern was statically correct but failed at runtime in `/$bunfs/root/` resolution. Another Claude session discovered it via the daemon logs after merge. Required reopening as #314 and a different fix approach.
2. **Jumped to implementing #286 fix before diagnosing** — code trace identified "Scenario A" and the fix was implemented before the user asked for diagnosis-first approach. Changes reverted. Plan was written to memory instead.
3. **Multiple sessions on same repo without branch coordination** — two sessions were active simultaneously, one tried `git reset --hard` on the other's working branch because it saw an unexpected branch state.
4. **`this.currentSpan` singleton race** — adversarial review caught it but it should have been obvious during design. Concurrent async HTTP requests sharing instance state is a classic mistake.
5. **Span lifecycle scope was unclear** — #293 was interpreted as "trace ID propagation only" when the intent was full OTel lifecycle. Resulted in a wasted PR (#302) and a follow-up redesign (#309). Gap was in the non-goals section of the issue being written too broadly.

## Patterns established

- **Diagnosis scripts → memory before implementing bug fixes**: when a P1 bug needs reproduce evidence, write the capture scripts and save the plan to `memory/` so it survives context clearing. Implement only after real data.
- **Worker bundling in compiled mode**: Bun `--compile` requires worker files as explicit entrypoints AND string path references (not `new URL()`). Use `--define=__COMPILED__=true` + conditional helper in `worker-path.ts` to support both dev and compiled modes.
- **Span table outbox pattern**: `exported_at` nullable timestamp (not `exported INTEGER`) — NULL = pending, non-NULL = when exported. Auto-prune by row count cap (not just time-based) to handle the no-exporter case.
- **`usage_stats` vs `spans` table**: usage_stats is the persistent event log (historical query), spans table is the short-lived export buffer (Prometheus-style drain). Keep separate.
- **Close PR, start fresh from main**: when adversarial review finds that all integration code will be rewritten, close the PR rather than fixing it. The primitives (trace.ts ID generation) stay; the integration layer starts over.
