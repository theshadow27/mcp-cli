## What worked well

- **Phase pipeline ran end-to-end for real work.** All 4 merged PRs went through `impl → triage → [review|qa] → [repair] → done` via `mcx phase run`. No legacy spawns, no manual gating.
- **Triage accuracy was solid.** Low scrutiny issues (#1410, #1411, #1412, #1414) routed to QA; high scrutiny (#1413, 3 packages) routed to review. Both paths were correct — the low scrutiny PRs that passed QA first try were genuinely clean, and #1413's reviewer caught real issues.
- **Micro-repair pattern saved money.** The #1413 reviewer self-repaired all 3 blocking findings (inverted comparator, untestable boundary, canary pre-release) in a single session ($1.33) instead of spawning a fresh opus repair. Updated the sticky review with a delta table.
- **Copilot inline comment audit caught real gaps.** PR #1427 had 2 valid Copilot comments (multi-line export default, side-effect imports) that the QA session missed. The 4-surface audit in run.md caught them before merge.
- **`done` phase correctly blocked on CI.** PR #1427 had a lint failure from the repair commit; `mcx phase run done` returned `ci_not_green` and refused to merge. Safety net worked.
- **Investigation issue (#1414) produced a fix.** The sprint plan expected it might close with "can't repro" — instead the impl session found the root cause and shipped a fix.

## What didn't work

- **QA passed #1410 before Copilot inline comments arrived.** The QA session ran local tests and posted `qa:pass`, but Copilot code review posted 2 valid inline comments after QA completed. The orchestrator's 4-surface audit caught this, but it required a repair cycle ($0.64) + lint fix ($0.10) + rebase ($0.10) that wouldn't have been needed if QA waited for Copilot. **Improvement:** consider adding a "wait for Copilot review" step in the QA phase, or documenting that QA should check inline comments before labeling.
- **Phase `done` doesn't auto-rebase.** Three PRs needed rebases after earlier PRs merged to main. Each required a one-shot sonnet spawn (~$0.10–0.13 each). The `done.ts` phase reports "conflicts" but the orchestrator has to manually spawn rebases. **Filed follow-up consideration for merge-queue (#1397).**
- **`qa:fail` label sticks after repair.** When #1412's repair pushed fixes, the `qa:fail` label remained. The QA phase saw the label and routed back to repair instead of re-running QA. Had to manually remove the label. **Improvement:** repair phase should clear `qa:fail` label on entry, or the phase script should check commit timestamps vs label timestamps.
- **Session `6586` stuck in `connecting` forever.** No logs, no result. Immediate `bye` per run.md. No root cause — likely a daemon race condition. Consistent with #1426.
- **Flaky `server-pool.spec.ts` SIGTERM 8000ms timeout** blocked #1412's CI. QA filed #1433. Required manual CI re-run + label swap.

## Patterns established

- **Phase pipeline is the default flow.** Sprint 36 proved it works for real customer bugs, not just toy issues. The orchestrator loop is: `mcx phase run <phase> → read JSON → execute spawn → mcx claude wait → repeat`.
- **Copilot comment audit is mandatory before merge.** 4-surface check (PR body, inline, review containers, linked issue) caught gaps that QA missed. This is orchestrator discipline, not automatable yet.
- **Auto-merge for trailing PRs.** When stopping a sprint early, arm auto-merge on in-flight PRs (`gh pr merge --squash --delete-branch --auto`) so they land without orchestrator attention.
