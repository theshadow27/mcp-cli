## What worked well

- **Sonnet for everything except implementation**: adversarial reviews, repairs, and QA all ran on sonnet. Saved significant opus quota while maintaining quality — sonnet reviews found real blocking bugs.
- **Orphaned sessions finished their work**: when the daemon restarted mid-sprint, 6 sessions lost their WS connection but had already pushed PRs. All PRs were recoverable.
- **Triage-driven pipeline**: low scrutiny issues went straight to QA, high scrutiny got adversarial review. All 3 high-scrutiny PRs (#644, #669, #653) had real bugs caught by review.

## What didn't work

- **Daemon restart mid-sprint**: user restarted daemon to test a fix, but the daemon was running stale code (pre-#618 well-known WS port). 6 sessions orphaned. Root cause: no pre-flight check that daemon is running latest build.
- **Lost session logs**: once orphaned sessions die and get `bye`'d, their logs become inaccessible. Had to reconstruct state from git history and PR listings.
- **Session seq reset**: after daemon restart, seq numbers reset to 0, which was confusing in monitoring.

## Patterns established

- **Pre-flight daemon check**: sprint run reference now includes a pre-flight step to verify daemon is running latest build before spawning sessions.
- **Wind-down daemon restart**: added explicit step to restart daemon after `bun run build` in wind-down, so next sprint starts clean.
- **Sonnet repairs for small fixes**: when adversarial review provides detailed findings, sonnet can handle the repair — opus not always needed.
- **Quota-aware sprint management**: when quota is tight, defer feature work and focus on completing the pipeline for in-flight issues rather than starting new ones.
