### Research: `--sdk-url` WebSocket protocol (#133 spike)
Discovered **[The-Vibe-Company/companion](https://github.com/The-Vibe-Company/companion)** which reverse-engineered the full `--sdk-url` protocol. Saved spec locally as `WEBSOCKET_PROTOCOL_REVERSED.md` (1358 lines). Key findings:
- `--sdk-url` makes Claude Code connect to YOUR WebSocket server as a client
- Bidirectional NDJSON: `system/init` (session_id, cwd, tools, model), `assistant`, `user`, `result`, `control_request` (can_use_tool)
- Full observation + control: send prompts, approve/deny tools, interrupt, change model, manage MCP servers live
- mcpd could sit as transparent proxy: tap → SQLite + forward to Companion for headed view
- `--headed` flag concept for per-session policy: headless (mcpd controls permissions) vs headed (forward to Companion, human approves in browser)
- Rewrote #133 as spike with 4 open questions: does `--sdk-url` still write `~/.claude/projects/` JSONL history? Separate `packages/claude`? Crash isolation via worker? Resume support? `--worktree` compat?

## Patterns established

- **Live test-drive is part of QA** — static code + unit test verification isn't enough for CLI commands. At minimum: run the happy path end-to-end before closing.
- **Spike issues for architectural unknowns** — when a design decision has 4+ open questions with real cost implications, write a spike issue with explicit acceptance criteria (answer each question with data) before implementation. Don't let enthusiasm for a clean design paper over unknowns.
- **Proxy-as-observer pattern** — mcpd-as-MITM for `--sdk-url` sessions (tap to SQLite, forward to Companion) is now the target architecture for #133. Permission routing policy: headless = mcpd decides, headed = forward to human via Companion.
