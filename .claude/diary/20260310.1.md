## Pipeline design

### What worked
1. **Plan mode for complex clusters** — ConfigWatcher had 6 overlapping issues. Plan-mode session read all 6, produced a holistic fix. User reviewed the plan, then a fresh session implemented it. Better than 6 separate PRs
2. **Sonnet for review/QA, opus for implementation** — review and QA don't need opus. Sonnet catches real bugs (secret exfiltration, timing issues, bare repo false positives) at much lower cost
3. **5+5 slot model** — 5 opus implementation slots + unlimited sonnet review/QA. Review and QA don't compete with implementation for slots
4. **Adversarial reviews caught real bugs every time** — secret exfil in dump, hardcoded "main" branch, try/catch placement making hangs worse, bare repo false positive. Worth the cost
5. **Combining related issues** — #461+#462+#463 (same method), #495+#493 (same file) as single PRs. Less overhead, more coherent fixes
6. **Dispute mechanism works** — #491 adversarial review claimed `--continue` won't work for worktrees. PR author disputed with evidence (`~/.claude/projects/` entries). Dispute was correct. Review was wrong

### What didn't work
1. **Plan-mode session couldn't exit plan mode** — spawned with only Read/Glob/Grep/Bash permissions, so ExitPlanMode/Write/Edit were blocked. Had to kill and spawn fresh. Fix: always include Write/Edit in plan sessions, or handle the permission dance
2. **QA merged with failing CI** — PR #513 had Bun segfault on Linux CI. QA rationalized "Bun bug" and merged anyway. Fixed by strengthening /qa command language
3. **One-session polling is wasteful** — sitting with 1 session running, polling every 30s, burns orchestrator tokens with no force multiplier. Need ≥3x parallelism to justify the manager overhead
4. **Daemon idle timeout during sprint** — daemon dies while orchestrator is between batches. Not catastrophic (auto-restarts) but causes transient failures
5. **Worktree accumulation** — 60+ dead worktrees by end of sprint. `--prune` command now exists but wasn't available until mid-sprint
6. **`--limit 50` instead of `--limit 200`** — missed 20 open issues in the board overview. Follow the skill spec

## Key learnings for orchestration

- **≥3 parallel sessions or don't bother managing** — the overhead of orchestration only pays off with parallelism
- **Sonnet review/QA doesn't count against implementation slots** — separate capacity pools
- **Dispute adversarial reviews with evidence** — they're sometimes wrong. Don't blindly repair
- **Plan mode needs Write/Edit permissions** to exit plan mode
- **Never rationalize CI failures** — if CI fails and you can't fix it, stop and report
- **`mcx version` vs `mcx --version`** — different things, both fine (rich status vs bare version string)
