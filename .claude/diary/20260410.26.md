## Worktree pollution investigation (#1109)

**Two root causes found:**

1. **cwd: undefined** — `createWorktree()` with `nativeWorktree: true` returned `{ worktree: name }` without `cwd`. The daemon spawned Claude with `cwd: undefined`, inheriting the daemon's working directory (main repo). Every file write leaked to the main tree. **Fix:** removed `nativeWorktree: true` from the headless spawn path.

2. **core.bare = true** — git flips `core.bare` to `true` during worktree add/remove operations. When this happens, `git status` and `git checkout -- .` break, and file operations behave unpredictably. The flag flipped 3 times during this sprint alone. **Fix (sledgehammer):** `fixCoreBare()` guard after every `git worktree add` and `git worktree remove`. This detects and corrects the flip but doesn't prevent it.

**Open question:** WHY does git set `core.bare = true` on a non-bare repo during worktree operations? Is it a git bug? A race condition from concurrent `git worktree add`? We applied a reliable bandaid but the root cause in git itself is uninvestigated. Worth filing upstream if it continues.

**Data point:** The 3493-line `.git/config` from 1161 stale branch tracking entries may be contributing. The oversized config file could be causing parse/write races during concurrent worktree operations.
