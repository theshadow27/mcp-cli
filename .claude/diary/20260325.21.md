# Sprint 21 Retro — 2026-03-25

## What happened

Sprint 21 landed the TUI feature pair (#81 server management, #82 registry browser) plus v1.0.0 release fix, 3 critical monitoring bug fixes, and 6 flaky test fixes. 17 PRs merged — 14 planned + 3 unplanned P1 daemon bugs discovered during execution.

## The monitoring bug incident

The agent command refactor (#913 from Sprint 20) broke three daemon features:
- **#983**: Sessions regressed to `init` state after completing work
- **#978**: `wait` returned stale data, didn't detect state changes
- **#985**: `wait --timeout` returned instantly instead of blocking

These made orchestration painful — had to poll with `sleep 120 && ls --short` instead of event-driven `wait`. All three were regression bugs from #913's routing changes.

**Root cause**: The agent command refactor changed how CLI args were parsed and forwarded to daemon tool calls. The `wait` timeout and event subscription weren't being passed through correctly. The `init` state bug was the daemon re-creating session records on CLI reconnect instead of updating them.

**Lesson**: High-scrutiny refactors that touch the CLI→daemon interface need integration tests that verify end-to-end behavior (spawn → work → wait detects idle), not just unit tests of individual functions.

## The reboot incident

A macOS reboot killed all 5 active sessions mid-sprint. Sessions go `connecting` → `disconnected` because the Claude CLI processes die and have no reconnection logic. Also found 133 stale worktrees accumulated across sprints 11-20 — cleaned all 140.

**Impact**: Had to respawn 5 sessions. Lost ~$100 in partial session work.

**Lesson**: The rebuild fixed it — the old daemon binary had the #983 bug which prevented new sessions from connecting. Always rebuild + restart daemon after merging daemon fixes, even mid-sprint.

## What worked

- **Adversarial review sticky comments** — the new pattern worked perfectly. Reviewers posted structured `## Adversarial Review` comments to PRs. Delta mode on re-review showed what was Fixed vs Not addressed. Repair sessions read PR comments first and had full context. No more "repair introduces new issues because it didn't know about previous findings."
- **`wait` is back** — after fixing #978/#985, the monitoring loop returned to event-driven `wait --timeout 30000`. Night and day difference vs polling.
- **Both TUI features shipped** — #81 (1133 additions, 2 review+repair rounds) and #82 (1889 additions, 2 review+repair rounds) both landed clean.
- **Filing issues immediately** — 4 bugs filed during sprint (#978, #981, #983, #985), all 3 P1s resolved in the same sprint.

## What didn't work

- **133 worktrees accumulated over 10 sprints.** Nobody noticed because `ls .claude/worktrees/` wasn't part of pre-flight. Add worktree count check to pre-flight.
- **`wait --timeout` was broken for the entire first half of the sprint.** The stale daemon from v1.0.0 was running while we merged wait fixes. Should have restarted daemon as soon as #983 merged instead of waiting for all sessions to complete.
- **$71 session for #983.** The init state bug was deep in the daemon state machine and the session struggled. Should have been timboxed — if a session hits $30, pause and evaluate whether the approach is right.
- **Disconnected sessions after reboot weren't detected.** `mcx claude ls` showed them as `disconnected` but `wait` returned instantly (because of #985). If `wait` had been working, it would have at least shown them.

## Patterns to keep

1. **Sticky review comments on PRs** — reviewers post, repairers read. Delta mode for re-reviews.
2. **Rebuild + restart daemon immediately** after merging daemon fixes, even mid-sprint.
3. **Clean worktrees at sprint boundaries.** Add to pre-flight checklist.
4. **File P1 bugs immediately and pull into current sprint** — don't defer daemon usability issues.
5. **Test `wait` after daemon restart** — quick smoke test: `time mcx claude wait --timeout 5000`.
