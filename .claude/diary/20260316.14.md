## What worked well

- **Mid-sprint pivot was smooth.** Swapping aliases for #754 took 2 minutes of sprint file edits. Pipeline absorbed the change with zero waste — no half-done alias work to revert.
- **Adversarial reviews continue to be essential.** Every high-scrutiny PR had real bugs: #754 had a process-hang-on-shutdown (CRITICAL), #705 had 10 findings including data loss on tab switch, #90 had a `-n` flag collision that would silently break 3 existing commands.
- **All 6 low-scrutiny PRs passed QA first try.** Triage model continues to be 100% accurate at separating safe from risky changes.
- **Batch 3 fillers ran in parallel with batch 2 reviews.** No idle opus slots at any point.

## What didn't work

- **PR #746 base branch problem.** #748 and #705 were merged into `feat/issue-704-plans-tab` instead of `main`. When it came time to merge the parent PR, massive conflicts (44 files). The merge session stalled. This happened because the QA sessions for these PRs targeted the feature branch, not main.
- **#705 needed 3 repair rounds.** First review: 10 findings. Second review after repair: 5 more findings (inflight lockout, refresh interface mismatch, string-as-mutex). Cost ~$14 total for one issue. The implementer didn't read the existing codebase deeply enough before writing.
- **#90 session used 15K tokens** — by far the most of any implementation. The `--verbose`/`--dry-run` feature touched many files and the session struggled with pre-commit hooks.

## Patterns established

- **P1 customer issues override sprint goals.** When #754 came in, we immediately dropped the planned alias work. Usability issues > feature work.
- **Feature branch merges are risky.** When PRs target a feature branch instead of main, the final merge becomes a conflict nightmare. Prefer direct-to-main PRs even for related work — the branch protection can handle it.
- **Two adversarial reviews for rewrites is non-negotiable.** Every second round found new bugs introduced by first-round fixes. This held true for #754, #748, and #705.
