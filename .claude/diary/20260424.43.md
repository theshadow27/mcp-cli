## What worked well

- **P0 serialization for `work-item-poller.ts`** — #1666 → #1667 → #1578 staged in that order; zero merge conflicts on the hot-shared file despite 4 PRs touching it. Plan's risk section called this out specifically and it held.
- **Opus for repair, sonnet for QA** — consistent pattern. Every opus repair session addressed its blockers in one round; when more Copilot findings arrived, they were new, not repeats.
- **Reviewer scope deferrals** — #1584's "remove `signal?` field + file follow-up for wiring" was the correct call; shipped minimum API and kept future wiring as #1727-adjacent. Saved ~2 repair rounds vs wiring AbortController through every alias context call site.
- **Pre-flight Copilot sweep** — sessions that did the 60s wait + `gh api /pulls/<n>/comments` check before declaring ready had significantly shorter QA cycles (eg. #1698, #1656 merged in 1 or 2 rounds; others without the sweep hit 3+ rounds).
- **CI re-run on flaky** — `gh run rerun <id> --failed` cleared the `core.bare=true repro` flake on #1705 without another commit. Cheap.
- **`defineMonitor` runtime critical bug (#1728) caught in QA** — saveAlias wasn't writing aliasType. Without QA's 4-surface discipline this would have merged non-functional. Worth the cycles.

## What didn't work

- **Copilot arriving *after* QA, repeatedly** — Copilot's review lag is ~30–120s, but QA often completes its sweep at the 60s mark. Result: `qa:pass` posted, then Copilot drops 3–8 new inline comments, forcing a `qa:fail` round. Hit this on #1710, #1711, #1713, #1723, #1717, #1709, #1731, #1732, #1733, #1741. **Fix filed as #1716 (meta)**; proposed gating qa:pass on an event like `waitForEvent({category: "review"})` with a cap, so the QA phase actually waits for Copilot's first pass.
- **Review + retro not run automatically after sprint** — user had to invoke `/sprint review and retro` explicitly in a second turn. The 379k-token context from the run was dropped by then; resuming cost a full cache miss. **Fix**: make `/sprint <N>` auto-chain run → review → retro. Writing up the fix in the sprint skill itself (below).
- **Copilot reviews posting new rounds of findings after each repair push** — Copilot re-scans the diff on every force-push. On larger PRs (#1713 MonitorRuntime), we hit 3 rounds of Copilot feedback; hit repair cap on round 3. Merged with one more repair round after explicit guidance "if findings are style nits, reply dismissing with follow-up; only blocker correctness triggers qa:fail". No memory update yet — may need a `/qa` skill refinement.
- **GitHub REST rate limit (5000/hour) hit at 1h20m into sprint** — all 16 spawned sessions + orchestrator share the same `gh auth` token, so parallel QA sweeps + inline-comment replies burned the budget fast. Recovery was automatic (resets hourly) but stalled ~3 minutes of work. No fix — the per-session `gh api` calls are necessary. Could mitigate with a daemon-side GitHub cache but that's a bigger project.
- **`mcx claude bye` deleted PR branches twice** — #1666 and #1667's session worktrees had the PR branch as the worktree branch. On `bye`, the safety check deleted the local branch (remote intact). Cosmetic but surprising. Probably worth filing: `bye` should leave PR branches alone when the worktree is a PR branch, not an ephemeral `worktree-<id>` branch.
- **Orchestrator meta files edited mid-sprint** — I touched `.claude/memory/feedback_sprint_bulk_and_cascade.md` (re-emphasizing the NO-for-loops rule after user flagged it) and `.claude/sprints/sprint-43.md` (start timestamp) while the sprint was running. These uncommitted changes sat on `main` until retro. Run.md says "Meta files are orchestrator + retro only" — which is what I did, but the window is longer than ideal. No regression, but worth noting.

## Patterns established

- **Repair round 3 escalation pattern**: when Copilot keeps posting new findings and we hit the 3-round cap, the orchestrator can spawn one more sonnet round with explicit "fix correctness, dismiss nits via reply" scope guidance. Used on #1584 (perf deferrals to #1727) and #1583 (round 3 final after 8 new findings). Saved two PRs from escalating to `needs-attention`.
- **Copilot sweep as a distinct impl step**: every session that reached "PR ready" was nudged to wait 60s, run `gh api /pulls/<n>/comments --jq '...'`, and fix+reply before declaring ready. When implemented as a post-push ritual, it cut round-1 QA fails roughly in half. Candidate for baking into `.claude/commands/implement.md` (but that's a meta file → retro-window change, not a worker-side instruction).
- **Small 1-surface repair sessions**: for QA-fails of "N Copilot threads, all minor", a sonnet repair with explicit thread IDs + fix descriptions completed in 1–3 minutes. Significantly cheaper than opus. Used on #1696 (6 threads), #1705 (2), #1677 (1), #1678 (2) — all closed in one round.
- **Rate-limit-tolerant event loop**: `mcx claude wait --timeout 260000` stays in cache; when the gh rate limit hits, sessions stall but the daemon keeps publishing events, so the orchestrator sees progress. Recovery is automatic at reset.
