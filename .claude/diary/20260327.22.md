## The idle detection breakthrough (#978)

The nerdsnipe on #978 found the root cause of sessions showing `active` instead of `idle`: the NDJSON result schemas (`ResultSuccess`/`ResultError`) were **too strict**. When the Claude CLI sent result messages with missing optional fields (`is_error`, `duration_api_ms`, `uuid`), Zod's `.parse()` threw, the exception propagated uncaught, and the state machine never transitioned to `idle`.

The fix adds a `ResultFallback` loose schema that matches any `type: "result"` message. The session then asked to audit ALL parse sites and found 3 more with the same bug class:
1. `handleInit` — `.parse()` throws → session stuck in `connecting` forever
2. `handleAssistant` — `.parse()` throws → token counts go stale
3. `handleControlRequest` — `.safeParse()` silent → permission requests swallowed

All being fixed in the same PR (#1047).

**Lesson:** Strict Zod schemas at protocol boundaries are a ticking time bomb. The Claude CLI is a moving target — field additions, renames, and optionality changes are expected. Parse sites should always use safeParse with fallback extraction and logging.

## The quota API spike (#1008)

First attempt was garbage — accepted "use session cost data" without questioning it. Cost data is broken (#1028) and meaningless for Max plans.

Second attempt found the real answer: rate limit headers exist on API responses (`anthropic-ratelimit-tokens-remaining`) but Claude CLI consumes them internally. However, we already receive `error: "rate_limit"` in NDJSON — just don't act on it. Filed #1043 for reactive handling.

Proactive path: use the OAuth token we already steal from the Keychain to make a lightweight API call and read the headers directly. Plan documented on the issue.

**Lesson:** Don't accept spike findings without scrutiny. Ask: "does this actually work for our use case?" The first session didn't know about Max plans or broken cost tracking.

## The cost tracking scare

Session costs showed $6994 and $1225 for normal sessions. Confirmed fake by checking the Anthropic account. Filed #1028. Root cause likely: WebSocket reconnection after laptop sleep causes cost accumulation corruption.

**Lesson:** Don't panic at large cost numbers — but also don't dismiss them. Verify against the actual account.

## What didn't work

- **Sessions showing `active` when done** — the very bug we were fixing (#978) bit us during the sprint. Had to poll with `ls` and check logs manually.
- **Bye'd the spike session too early** — lost the #1008 context before the user could ask follow-ups. New rule: don't bye investigation sessions until user signs off.
- **Accepted bad spike findings** — first #1008 attempt recommended using broken cost data. Should have cross-referenced with known issues.
- **`for` loops in bash trigger manual approval** — keeps causing laptop sleep interruptions. Must use separate parallel bash calls instead.
- **Runaway session costs ($6994 display)** — daemon cost tracking is unreliable after sleep/reconnect. Can't trust the numbers.

## Patterns to keep

1. **Nerdsnipes with detailed instructions** — the investigation guide format works. Be specific about what to check, what files, what to log.
2. **Don't bye investigation sessions** — leave them alive for follow-up questions.
3. **Audit the whole subsystem** — when you find a bug class, check every instance. The #978 audit found 3 more.
4. **Log Bun segfaults on #1004** — every CI failure with a bun.report URL gets documented.
5. **Scrutinize spike findings** — ask "does this work for Max plans?" and "is the data source reliable?"
