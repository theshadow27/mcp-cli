## What worked well

- **WIP-commit resurrection saved 7 worktrees from OOM oblivion.** When the user suggested adding a synthetic WIP commit to bypass `mcx claude resume`'s false-positive "already merged" check, all 8 dead workers came back. 4 of them had real uncommitted edits (Frank/#2060 had 4 modified files, Eve/#2052 had 5, Hank/#2071 had 3, Dave/#2075 had 4 — those would have been re-done from scratch otherwise). The other 4 needed `--allow-empty`. Every resumed session pushed a clean PR within minutes. This is the load-bearing pattern for crash recovery now — the bug report itself (#2082) ranks "spawn fresh with transcript-as-context" as the inferior workaround.

- **Reviewer self-repair on #2073 (June → 3 fixes + Copilot's runtime-throw bonus in one commit) cycled fast.** When the adversarial review found 1 blocker + 2 should-fix and Copilot independently flagged the same blocker plus the satisfies-never trap, the `send-the-reviewer-back-to-fix-its-own-findings` pattern produced commit `0a8ccba` with all 4 resolved in 6 turns. No fresh opus repair needed.

- **Bun 1.3.14 bump cascade-closed 3 issues for free.** The plan flagged #2055/#2068/#2058 as "gated on #2072 — re-test under the bump." Alice ran the re-test loop on her own initiative as a bonus and closed both segfault issues; Frank picked up #2058's verification cleanly post-recovery. That's 3 issues resolved with no incremental implementation cost.

- **Triage's `goto qa` for low-scrutiny work routed correctly on 6/9 PRs.** Where the metric (src churn + risk areas) was loose, the orchestrator caught it (see #2073 below); where it was right, the QA-only path saved a full adversarial-review round per PR.

## What didn't work

- **Mid-sprint OOM (Ghostty 458 GB)** — the most disruptive single event of the sprint. All 11 active worker sessions disconnected when the user's terminal hit a memory wall. Root cause is most likely buffer accumulation in Ghostty itself from the orchestrator's repeated `mcx tracked --json | jq ...` invocations, since `mcx tracked` carries 40+ stale done-phase rows from sprints 14-52 (filed #2091 to auto-prune). The crash was not a daemon or worker bug — but the orchestrator behavior (verbose JSON dumps × hundreds of poll iterations) is fixable.
  - **Fix made**: filed #2091 to auto-prune `phase=done` work items older than 7 days; will land in sprint 56.
  - **Process change**: when polling state during the run, prefer `--phase` filters or single-item `work_items_get` over the full `--json` dump.

- **`mcx claude resume` blocks recovery of branches with 0 commits ahead of main (#2082).** Every dead worker had this — either a worktree-`*` scratch branch (review/QA workers don't commit) or a feature branch where the impl session was killed before its first commit. `git branch --merged` reports both as "merged", and resume refuses. We worked around it with the WIP-commit hack but the proper fix is to either (a) skip the merge check when a JSONL transcript exists for the worktree, or (b) gate it behind a `--force` flag. This bug is load-bearing for any sprint that hits a transient failure (daemon restart, OOM, network blip).
  - **Fix made**: filed #2082 with reproducer; ranked priorities in the comment thread (impl workers affected too, not just review/QA).

- **Triage downgraded #2073 to low scrutiny by line count (18 LOC, 1 src file).** The framework refactor that 2 follow-up modules depend on got the same treatment as a docstring fix. I manually forced `phase=review` because the plan explicitly tagged it `high (gated)`. Without that manual override, the 1 blocker + 3 should-fix findings would have been deferred to post-merge follow-ups.
  - **Fix made**: triage's scrutiny model should consider not just churn but also "is this a framework/dispatcher/orchestration file that downstream PRs build on." File a sprint-56 issue for the heuristic upgrade.

- **5 Copilot inline comments arrived post-qa:pass on #2085 (#2052), 3 more on #2086 (#2075), 1 more on #2080 (#2073).** The race between auto-merge queuing and Copilot's per-PR review pipeline means the orchestrator can mark a PR `qa:pass`, queue the merge, and only see substantive findings minutes later. On #2086 the PR had already merged when 3 polish-grade comments landed (filed #2090 for follow-up). On the others we caught it in time and the QA workers were sent back to address — but the pattern is fragile.
  - **Fix made**: orchestrator now waits ~60s after qa:pass before merging if Copilot review is still pending. (Not yet rule-text — captured here, formalize in run.md if it recurs.) Alternatively: don't merge until the Copilot pipeline `state == COMPLETED`.

- **Quota hit 100% after the recovery cascade.** Resuming 6 sessions + their downstream QAs after the OOM burned through the 5-hour budget. #2069 and #2022 (deferred to sprint 56) were both in-flight before crash; the resumes would have been cheap, but we hit the cap before getting there.
  - **Fix made**: none structural — accept that an OOM mid-sprint will cost the rest of the quota window. Sprint 56 inherits both as Bucket-1 candidates; Iris's worktree for #2022 is intact for `mcx claude resume` fast-track.

## Patterns established

- **WIP-commit resurrection** — the canonical recovery for any session-lost-but-worktree-intact scenario. Once #2082 lands this becomes obsolete; until then, use it.
- **Reviewer self-repair scales beyond 3 findings when the reviewer has full diagnosis.** June handled 3 review + 1 Copilot finding in one commit. The threshold in run.md ("1-3 contained edits") can probably be relaxed when each finding has explicit file:line + fix description.
- **Cluster PRs (one PR closes N related issues) work cleanly** — Alice's #2087 closed #2060 + #2061 + #2062 in one squash. No special tracking needed; just attach the lead issue to the work item and let the merged-PR cleanup module untrack the bundled issues.
